## What This Proves

- 2 socket.io clients connect via Traefik (`localhost:8000`) and pin to different app replicas via the `nh_sticky` cookie issued by Traefik's sticky-session load balancer.
- A `POST /api/_test/emit-fanout` HTTP request is routed to one replica; that replica calls `io.emit('test:fanout', payload)`.
- The `@socket.io/redis-adapter` (Plan 37-01) republishes the emit on Redis pub/sub channels (`socket.io#/#`).
- The OTHER replica (where client B is pinned) receives the pub/sub message, dispatches to client B locally, and client B's listener fires within the 5 s budget.
- Test assertion `expect(data).toMatchObject(payload)` matches.

This closes the locked gate for **INFRA-04** (sticky sessions + cross-replica WebSocket coherence) and the runtime-only portion of **INFRA-02** (LB distribution to multiple replicas).

The last commit is the substantive fix: `socket.io-client` v4 in Node.js does not manage cookies across HTTP requests (browsers do, Node does not). With sticky-session load-balancing in front of socket.io polling, the initial GET handshake gets a `sid` + `nh_sticky` cookie back, but the follow-up POST goes WITHOUT the cookie, gets routed by Traefik to a different replica that doesn't know the `sid`, and returns "xhr post error". Switching `transports: ['polling', 'websocket']` → `transports: ['websocket']` sidesteps this — WebSocket is a single TCP connection, sticky pinning happens once at upgrade time and the socket stays pinned. Production browsers don't have this issue (they manage cookies natively). The Redis-adapter fanout proof is transport-agnostic anyway — the adapter operates on Socket.IO message pub/sub, not engine.io transport.
