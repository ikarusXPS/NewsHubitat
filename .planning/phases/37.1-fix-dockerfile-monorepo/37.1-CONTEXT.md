<decisions>
## Implementation Decisions

### Package manager (Plan 01)
- **D-01 [LOCKED]** — Use **pnpm via corepack**. Pin to `pnpm@10.32.1` (the version `pnpm install` last used in this repo per the lockfile metadata; CI workflows also pin major `version: 10`). In the Dockerfile, `RUN corepack enable && corepack prepare pnpm@10.32.1 --activate` before any `pnpm` invocation. Rationale: matches local + CI; no need to add pnpm as an explicit npm install step; avoids drift.

### What to copy in the deps stage (Plan 01)
- **D-02 [LOCKED]** — Copy these files BEFORE running `pnpm install` so docker-layer caching kicks in only when manifests change:
  - `pnpm-workspace.yaml` (root)
  - `pnpm-lock.yaml` (root)
  - `package.json` (root)
  - `apps/web/package.json`
  - `packages/types/package.json`
  - `apps/web/prisma/schema.prisma` (so a `postinstall` hook that runs `prisma generate` has the schema available — though this repo doesn't currently rely on a postinstall, defensive)

  Subsequent workspace packages (e.g., a future `packages/api-client`) MUST be added to this list when added to `pnpm-workspace.yaml`. Document this in a comment.

### Install scope (Plan 01)
- **D-03 [LOCKED]** — Deps stage runs `pnpm install --frozen-lockfile --filter @newshub/web...` (the `...` includes all packages `@newshub/web` depends on transitively, currently `packages/types`). Builder stage reuses the deps stage's `node_modules`. Runner stage runs `pnpm install --frozen-lockfile --filter @newshub/web... --prod` — production deps only — to slim the runtime image. Rationale: `@newshub/web` is the only deployable; `packages/types` is a build-time dep; pulling the whole workspace would inflate the image with toolchain-only deps from any future package.

### Prisma generate (Plan 01)
- **D-04 [LOCKED]** — In the builder stage, run `pnpm --filter @newshub/web exec prisma generate` (NOT `npx prisma generate` from root). This invokes Prisma 7 with `apps/web/prisma/schema.prisma` and emits the client to `apps/web/src/generated/prisma/`. Run again in the runner stage AFTER the prod-only install to ensure the generated client matches the runtime platform (linux-musl on Alpine).

### Build invocation (Plan 01)
- **D-05 [LOCKED]** — Builder stage runs `pnpm --filter @newshub/web build` which the workspace's `package.json` resolves to `npm run build:frontend && npm run build:server` → `vite build` (frontend → `apps/web/dist/`) + `tsup` (server → `apps/web/dist/server/index.js`). Do NOT run `pnpm -r build` at the root — that would attempt to build `packages/types` which only emits TypeScript declarations and is already produced by Prisma's generate step.

### CMD path (Plan 01)
- **D-06 [LOCKED]** — Runner CMD is `["node", "--import", "./apps/web/server/instrument.mjs", "apps/web/dist/server/index.js"]`. Mirrors `apps/web/package.json` start script (`node --import ./server/instrument.mjs dist/server/index.js`) but with monorepo-relative paths from `/app` (the WORKDIR). The `--import` flag is required for Sentry instrumentation pre-loading; dropping it silently breaks error reporting in production.

### Healthcheck (Plan 01)
- **D-07 [LOCKED]** — Keep `HEALTHCHECK CMD wget -q --spider http://localhost:3001/api/health/db || exit 1`. The `/api/health/db` endpoint exists on `@newshub/web`'s server and returns 200 when Prisma can `SELECT 1`. Do NOT switch to `/api/ready` here — `/api/ready` is the load-balancer-facing readiness probe that returns 503 during graceful drain (see Phase 37 plan-05); a `HEALTHCHECK` returning unhealthy mid-drain would cause Docker/Swarm to kill the container before drain completes. `/api/health/db` is the liveness probe and remains 200 until process exit.

### Chromium / Puppeteer block (Plan 01)
- **D-08 [LOCKED]** — Preserve the existing Chromium + nss + freetype + harfbuzz + ca-certificates + ttf-freefont apk install verbatim. Required for D-03 (Puppeteer) per the Phase 35 commit history. Keep the `PUPPETEER_SKIP_CHROMIUM_DOWNLOAD=true` and `PUPPETEER_EXECUTABLE_PATH=/usr/bin/chromium-browser` env vars.

### Non-root user (Plan 01)
- **D-09 [LOCKED]** — Keep the `nodejs` user creation and the `USER nodejs` directive at the bottom of the runner stage. `chown -R nodejs:nodejs /app` after copying built artifacts (the `--chown=nodejs:nodejs` shorthand in the existing `COPY --from=builder` is fine but every COPY into the runner must include it). Required for D-05 (security: containers run non-root).

### .dockerignore (Plan 01)
- **D-10 [LOCKED]** — Verify root `.dockerignore` excludes `node_modules`, `.git`, `.planning`, `.claude`, `e2e-stack/node_modules`, `apps/web/node_modules`, `packages/*/node_modules`, `dist`, `apps/web/dist`, `coverage`, `*.log`. Add any missing patterns. The build context transferred 457 MB on the first failed run — likely because `node_modules` got pulled in from somewhere; trim aggressively.

### Plan 02 verification flow
- **D-11 [LOCKED]** — Plan 02 is `autonomous: false` because Task 1 ("re-run `bash e2e-stack/run-fanout-test.sh`") requires Docker Desktop on the operator's host. The plan's Tasks 2-5 (close-out doc edits) ARE autonomous and can run after the operator returns "approved" with the test output pasted. ## Specific Ideas

- The first failed build on 2026-04-29 transferred 457 MB of build context. Likely caused by missing `.dockerignore` entries for `apps/web/node_modules` (workspace packages have their own `node_modules`). Plan 01 must verify the dockerignore exclusions and trim if needed.

- Phase 37 plan-05 added `@godaddy/terminus@^4.12.1` and the `/api/ready` endpoint. The rewritten Dockerfile MUST install this dep (it's already in `apps/web/package.json` deps; pnpm install will pick it up). No special handling needed but worth a verify-line in Plan 01.

- The operator's failed run hit cache HIT for the chromium apk install (28 sec saved on retry); after Plan 01 lands, the first build will rebuild the chromium layer once because `apk add --no-cache` is not deterministic with cache busting if Alpine's APKINDEX changed. Acceptable.

- Plan 02 has a tight ~10-line code change (the four reconciliation edits) but its real value is in the verification — orchestrator surfaces the human-verify checkpoint, operator runs the test, returns approval or failure output.

</specifics>

<deferred>
## Deferred Ideas

- **Multi-stage cache mounts** (`RUN --mount=type=cache,target=/root/.local/share/pnpm/store`) — would speed up rebuild but requires BuildKit and adds complexity. Skip — the existing layered cache is good enough.
- **Distroless runner** — could shrink the runner image significantly, but loses the ability to `apk add chromium`. Skip — Puppeteer dep is hard.
- **Switch HEALTHCHECK to a curl-based probe** — `wget` works fine on Alpine. Skip — no benefit.
- **Bundle CSS/JS hashing in build args** — vite already handles this. Skip.
- **k6 smoke + Grafana dashboard verification** (HUMAN-UAT test #2) and **rolling-update drain test** (HUMAN-UAT test #3) — those need a real Swarm and remain pending. Out of scope for 37.1.

</deferred>
