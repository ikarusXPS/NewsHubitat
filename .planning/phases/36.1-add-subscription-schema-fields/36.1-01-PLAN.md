Required Prisma field declarations (the exact lines to add to the User model):
```prisma
// Phase 36 / 36.1: Stripe Subscription Fields (PAY-01)
stripeCustomerId       String?   @unique
stripeSubscriptionId   String?   @unique
subscriptionTier       String?   // 'FREE' | 'PREMIUM' | 'ENTERPRISE' (typed in server/config/stripe.ts)
subscriptionStatus     String?   // 'ACTIVE' | 'PAST_DUE' | 'CANCELED' | 'PAUSED'
subscriptionEndsAt     DateTime?
```

Decision (Claude's discretion, justified inline):
- **Field-level `@unique` over `@@unique([...])` block**: Both satisfy `findFirst({ where: { stripeCustomerId } })`, but `@unique` produces a cleaner generated client (`prisma.user.findUnique({ where: { stripeCustomerId } })` becomes available too), and matches the existing pattern in this same User model (`googleIdHash String? @unique`, `githubIdHash String? @unique`). Consistent with codebase conventions.
- **String fields, no Prisma enum**: `SubscriptionTier` is a TypeScript union in `server/config/stripe.ts`. The 36-01 SUMMARY claimed Prisma enums were added but they weren't, AND the existing service code casts via `as SubscriptionTier`. Plain `String?` in the DB matches the existing pattern in this codebase (e.g. `Comment.flagReasons` is `Json`, not enum). No type safety lost â€” the cast happens at the service boundary.
- **All fields nullable**: The service treats missing/null subscription as FREE tier (see `subscriptionService.ts:147` â€” returns `{ tier: 'FREE', status: 'ACTIVE' }` when user not found). Backfilling existing rows to defaults is therefore unnecessary; nullability is the correct expression of "no Stripe subscription yet".
