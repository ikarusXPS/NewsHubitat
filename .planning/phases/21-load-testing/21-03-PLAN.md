  <action>
Per D-13: Run smoke test (1 minute, 10 VUs) to validate k6 setup before attempting full 10k VU load test.

**Prerequisites verified by Checkpoint 1:**
- k6/data/users.json exists (100 test users)
- Server is running at http://localhost:3001

**Execute smoke test:**
```bash
# Run smoke test scenario
K6_BASE_URL=http://localhost:3001 k6 run k6/load-test.js --env K6_SCENARIO=smoke
```

**Expected behavior:**
1. k6 loads test script and validates configuration
2. Smoke scenario starts: 0 VUs ramp to 10 VUs over 30 seconds
3. Each VU executes user journey: login → browse news → AI (5% chance) → bookmark (10% chance)
4. Smoke scenario holds at 10 VUs for 30 seconds (1 minute total)
5. k6 evaluates thresholds (looser for smoke than load per D-25)
6. handleSummary() generates summary.html and summary.json
7. Test completes with exit code 0 (pass) or 1 (threshold breach)

**Success criteria:**
- summary.html and summary.json files created in working directory
- Test exits with code 0 (all checks and thresholds pass)
- No connection errors (server reachable)
- No authentication failures (test users valid)

**If test fails:**
- Check server logs for errors
- Verify k6/data/users.json has valid credentials
- Check rate limiting (429 responses are expected, not failures per D-07)
- Review summary.html for detailed failure information
  </action>

  <verify>
    <automated>
ls summary.html summary.json 2>&1 | grep -c "summary" || echo "0"
    </automated>
  </verify>

  <acceptance_criteria>
- summary.html exists and contains k6 test report with charts
- summary.json exists and contains metrics object
- k6 exit code is 0 (test passed)
- summary.json contains metrics.http_req_duration with p95 and p99 values
- summary.json contains metrics.http_req_failed with rate value < 0.01 (1%)
- summary.json contains metrics.checks with passes > 0
- No error messages about missing users.json or connection refused
  </acceptance_criteria>

Summary reports generated:

  <how-to-verify>
**Step 1: Review HTML report**
Open summary.html in browser:
```bash
# macOS
open summary.html

# Linux
xdg-open summary.html

# Windows
start summary.html
```

**Check for:**
- [ ] **Checks section**: "Checks passed" should be > 95% (some 429s expected)
- [ ] **HTTP Requests section**: Request rate ~5-10 req/s during peak
- [ ] **Performance section**: p95 latency < 500ms for most requests
- [ ] **Thresholds section**: All thresholds show ✅ green checkmarks

**Step 2: Verify metrics in JSON**
```bash
# Check p95 latency
jq '.metrics.http_req_duration.values["p(95)"]' summary.json
# Should be < 500ms (smoke test looser threshold)

# Check error rate
jq '.metrics.http_req_failed.values.rate' summary.json
# Should be < 0.01 (1%)

# Check total iterations
jq '.metrics.iterations.values.count' summary.json
# Should be > 50 (10 VUs × ~5-10 iterations each)
```

**Step 3: Verify user journey execution**
Check that all endpoints were hit:
```bash
# Count endpoint tags
jq '.metrics.http_req_duration.values | keys | map(select(contains("endpoint:")))' summary.json
# Should include: endpoint:auth, endpoint:news, endpoint:ai, endpoint:bookmarks
```

**Step 4: Check for expected 429 rate limit responses**
```bash
# These are EXPECTED and NOT counted as errors (D-07)
grep -i "429" summary.html
# Should show some 429 responses (normal behavior under rate limiting)
```

**Expected outcome:**
- ✅ Smoke test passes with all thresholds green
- ✅ Summary reports show realistic traffic pattern
- ✅ No unexpected errors (connection refused, auth failures, etc.)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| k6 Client → Local Server | Load test client sends HTTP requests to localhost:3001 |
| Database → Test Users | Test users created in production database (flagged with loadtest* pattern) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-21-09 | Information Disclosure | users.json on developer machine | accept | File is gitignored; developers responsible for local file security |
| T-21-10 | Tampering | Test users in production database | mitigate | Test users easily identifiable (loadtest* email pattern); can be bulk-deleted via SQL |
| T-21-11 | Denial of Service | Local smoke test overwhelming local machine | accept | Smoke test limited to 10 VUs (low resource usage); full test runs in CI only |
</threat_model>
