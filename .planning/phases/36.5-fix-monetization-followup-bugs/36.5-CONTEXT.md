Close two non-blocking defects surfaced during Plan 36-05's live human-verify (2026-04-28, ikarus.nbg@gmail.com on Stripe test sandbox) that did not block PREMIUM end-to-end functionality but degrade observability and produce inconsistent UI:

2. **Bug B (PAY-04):** Sidebar tier-badge is decoupled from real subscription tier. Observed: IKARUSXPS sidebar showed "Premium" while DB had `subscriptionTier=FREE`. Root cause is twofold:
   - The `showPremiumBadge` boolean column on User is a dead vanity flag (no server code reads or writes it; `@default(true)` means every user has `showPremiumBadge=true`).
   - `apps/web/src/components/Sidebar.tsx` lines 293‚Äì294 hard-code the literal text `Premium` with no conditional on user state.

NOT in scope: re-running the 36.4-04 D-09 probes against the corrected `authMiddleware ‚Üí aiTierLimiter` middleware chain (queued separately as Phase 36.6 candidate per STATE.md Deferred Items row 76).

<decisions>
## Implementation Decisions

### showPremiumBadge column (criterion #3)
- **D-01 [LOCKED]** ‚Äî Drop the `showPremiumBadge` column from the User model. Source-of-truth for the "show premium badge" UI render is `User.subscriptionTier` (PREMIUM or ENTERPRISE). User confirmed via plan-phase gate: option (a) "drop column" preferred over option (b) "keep with sync invariant". Rationale: no server code reads or writes `showPremiumBadge` (verified via repo-wide grep ‚Äî zero hits outside generated/prisma/), so there is no data to migrate; column is pure dead weight that already produced a UI drift bug. Migration is a non-destructive `prisma db push` (drops column; no row rewrites).

### Sidebar premium label (criterion #4)
- **D-02 [LOCKED]** ‚Äî `apps/web/src/components/Sidebar.tsx:293-294` must change from a hard-coded `Premium` text node to a conditional render keyed on `user?.subscriptionTier`. Render the label only when `subscriptionTier === 'PREMIUM' || subscriptionTier === 'ENTERPRISE'`. The component currently has no prop for subscription tier; it must read from `useAuth()` (see D-03).

- **D-03 [LOCKED]** ‚Äî Extend the frontend `User` interface in `apps/web/src/contexts/AuthContext.tsx` (lines 6‚Äì21) to include `subscriptionTier: 'FREE' | 'PREMIUM' | 'ENTERPRISE'`. Server-side `/api/auth/me` (`apps/web/server/routes/auth.ts:121-136` ‚Üí `authService.getUserById` line 162) already returns the full user row minus `passwordHash` via spread, so `subscriptionTier` is already on the wire ‚Äî only the client-side type needs to widen and Sidebar needs to read it. (Other components already cast: `Pricing.tsx:151`, `AIUsageCounter.tsx:23` use `(user as { subscriptionTier?: string }).subscriptionTier` ‚Äî that cast pattern can be removed once the type is widened, but doing so is out of scope unless the planner finds it necessary to satisfy criterion #4.)

### customer.subscription.created handler (criterion #1)
- **D-04 [LOCKED]** ‚Äî The fix has two layers:
  - **Layer 1 (logging):** `apps/web/server/routes/webhooks/stripe.ts:72` currently calls `logger.error('[Webhook] Processing error:', err);`. When `err` is a non-Error object or has an empty `message`, this produces the empty-error log observed in 36-05. The handler MUST serialize errors to a structured payload ‚Äî at minimum `{ message, name, stack, code }` plus `eventId` and `eventType` from the outer scope ‚Äî so even unusual errors render with diagnostic content.
  - **Layer 2 (root-cause diagnosis):** `customer.subscription.created` and `customer.subscription.updated` both route to `handleSubscriptionUpdated` (`apps/web/server/services/stripeWebhookService.ts:110-113`). When `customer.subscription.created` fires AFTER `checkout.session.completed`, the handler's flow is: `findUserBySubscriptionId(subscription.id)` ‚Üí if found, `updateUserSubscription(userId, subscription, customerId)`. The plan must add an integration test that reproduces the failure path (per criterion #2) and use it to drive a real fix in the handler. Hypothesis from 36-05-SUMMARY:148: race condition with concurrent `checkout.session.completed`, missing field on subscription object, or a Prisma write conflict. The plan must NOT short-circuit by silencing the error; it must diagnose and fix.

### Integration test (criterion #2)
- **D-05 [LOCKED]** ‚Äî Add a vitest integration test at `apps/web/server/services/stripeWebhookService.test.ts` (or extend if it exists) covering:
  - (i) `checkout.session.completed` runs first and populates User ‚Üí `customer.subscription.created` arrives second ‚Üí handler completes cleanly with no error log
  - (ii) `customer.subscription.created` arrives in isolation (no prior checkout) ‚Üí User row updated as expected
  - Use prisma mock (existing pattern; see `subscriptionService.test.ts` for analog). Test must fail BEFORE the handler/logging fix and PASS after.

### Anti-pattern guards (criterion #6)

(The `--accept-data-loss` flag is required because dropping a column is technically a destructive op, even though no application code reads the column.) The regenerated client lives at the gitignored path `apps/web/src/generated/prisma/` (per `.gitignore:31`) and is not committed.

<specifics>
## Specific Findings (verified during plan-phase context-gathering, 2026-04-28)

1. `showPremiumBadge` has zero non-generated references in the codebase. Grep targets: `apps/web/src/**` (excluding `generated/**`), `apps/web/server/**`, `apps/web/prisma/**`. Only hit outside the column declaration is `.planning/` documentation. **Implication:** column is fully dead ‚Äî drop is a one-edit operation in `schema.prisma`, no service-layer cleanup needed.

2. `Sidebar.tsx` has NO existing import of `useAuth` or any user-state hook. The component is currently stateless w.r.t. authentication. Adding the conditional render requires either:
   - importing `useAuth` from `../contexts/AuthContext` and reading `user?.subscriptionTier`, OR
   - hoisting the prop from a parent (Header/AppShell) ‚Äî riskier given Sidebar is mounted in many routes.
   The hook-import pattern matches `Header.tsx` and other authenticated views; planner should pick that.

3. `SubscriptionBadge` component (`apps/web/src/components/subscription/SubscriptionBadge.tsx`) ALREADY exists with proper tier-prop API and gates render to non-FREE tiers. Reusing it in Sidebar is the cleanest fix path ‚Äî pass `tier={user?.subscriptionTier ?? 'FREE'}` and let the component's existing `if (tier === 'FREE') return null` handle the conditional. **Strongly recommended over a custom inline label.**

4. `/api/auth/me` already returns `subscriptionTier` on the wire (server side returns the full User row minus `passwordHash`); only the client-side TypeScript `User` interface is too narrow. Widening it is a 1-line change.

5. The empty-error log in Bug A almost certainly originates in `handleSubscriptionUpdated` (`stripeWebhookService.ts:182`) when `customer.subscription.created` fires after `checkout.session.completed` had already linked the user. The integration test from criterion #2 is the diagnostic instrument that will reveal the real handler bug.

6. Test-suite baseline: `pnpm test:run` is 1304/1304 PASS as of commit `4f0b4d2`. After this phase: at minimum 1305+ (one new integration test from D-05); planner may add more.

</specifics>
