## Enum Migration

### Q1 — Enum casing

| Option | Description | Selected |
|--------|-------------|----------|
| UPPERCASE (FREE, PREMIUM, ENTERPRISE) | Matches existing String? values and the spec in 36-CONTEXT.md. Lowest churn for subscriptionService.ts and tests. | ✓ |
| lowercase (free, premium, enterprise) | Matches existing ApiKeyTier convention. Cleaner Prisma idiom but requires renaming every consumer site, every test fixture, all `?? 'FREE'` fallbacks. | |
| Keep TS union; do NOT add Prisma enums | Override roadmap success criterion #3. Stay on 36.1's path. Requires a roadmap edit/decision-record. | |

**User's choice:** UPPERCASE.
**Notes:** Recommended option taken. Internal inconsistency with `ApiKeyTier { free, pro }` is accepted as known and not worth the consumer-code churn to harmonize.

### Q2 — Backfill / type-conversion strategy

| Option | Description | Selected |
|--------|-------------|----------|
| `prisma db push --accept-data-loss` in dev | Treat dev DB as disposable — same approach 36.1 used. DB has no real subscription data yet. Adds a PRODUCTION-MIGRATION.md note for future prod cutover. | ✓ |
| Hand-crafted SQL migration file + prisma db pull | Resurrect the migrations workflow with USING CAST text-to-enum. Heavier process but reusable for production. | |
| Drop columns first, then re-add as enum, then push | Two-step push. Preserves other dev DB data but loses tier/status assignments. | |

**User's choice:** db push --accept-data-loss.
**Notes:** Existing migrations folder is sqlite-locked legacy; project is on db-push workflow. Production migration deferred to its own phase (flagged in CONTEXT.md `<deferred>`).

### Q3 — TS union fate in stripe.ts

| Option | Description | Selected |
|--------|-------------|----------|
| Replace with re-export of Prisma enum | `export { SubscriptionTier, SubscriptionStatus } from '@prisma/client'`. Single source of truth. Zero call-site changes if casing matches. | ✓ |
| Delete the TS unions; import directly from @prisma/client at every call site | Cleaner but requires touching ~10 files to change import paths. | |
| Keep both side-by-side | Risks drift if someone adds a tier to one and not the other. | |

**User's choice:** Re-export from Prisma client.
**Notes:** Combined with Q1's UPPERCASE choice, this means **zero consumer-code changes** are needed for the enum migration.

## New Model Relations + Cardinality

### Q4 — ReferralReward FKs

| Option | Description | Selected |
|--------|-------------|----------|
| Both relations to User with onDelete: Cascade | Two named relations: ReferralRewardReferrer and ReferralRewardReferred. Referential integrity guaranteed. | ✓ |
| Plain String columns, no relation | Simpler schema; matches ShareClick.createdBy. Cost: orphan rows possible. | |
| FK on referrerId only; referredId stays plain String | Middle ground. Referrer is the key relation. | |

**User's choice:** Both as relations with onDelete: Cascade.
**Notes:** Two indexes added (`@@index([referrerId])`, `@@index([referredId])`). Cascade matches TeamMember.user pattern.

### Q5 — StudentVerification FK

| Option | Description | Selected |
|--------|-------------|----------|
| Prisma relation to User with onDelete: Cascade | User deletion cascades. Consistent with TeamMember.user pattern. | ✓ |
| Plain String userId, no relation | Inconsistent with the codebase. | |

**User's choice:** Prisma relation, Cascade.
**Notes:** `studentVerifications StudentVerification[]` added to User. Indexes on userId and status per spec.

### Q6 — referredBy semantics

| Option | Description | Selected |
|--------|-------------|----------|
| User.id of the referrer (self-relation) | referredByUser User? @relation("UserReferrer"...). Matches 36-CONTEXT.md intent. | ✓ |
| Free String, the referral code that was used | Decouples from User identity; lookup-by-code via referralCode column. | |
| Both: code AND FK to User | Belt and suspenders. Adds a column. | |

**User's choice:** User.id self-relation.
**Notes:** `onDelete: SetNull` so deleting a referrer doesn't cascade-delete their referred users. `referralCode String? @unique` stays as the *referrer's* code other users enter at signup.
