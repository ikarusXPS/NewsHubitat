## Decisions Made

- **Schema flag override on db push (`--schema=prisma/schema.prisma`):** The root `prisma.config.ts` configures `schema: "prisma/schema.prisma"`, which (relative to cwd `apps/web`) resolves to `apps/web/prisma/schema.prisma` — but only **after** noticing the first attempt without the explicit flag actually loaded `D:\NewsHub\prisma\schema.prisma` (a stale duplicate at the repo root that lacks ApiKey). Adding the explicit flag pinned the canonical apps/web schema. See "Deviations from Plan" below.
- **Import path `'../../src/generated/prisma/client'`:** Plan 03 specified `'../../src/generated/prisma'` but Prisma 7's `prisma-client` generator emits no `index.ts` — directory imports do not resolve. The codebase's existing `apps/web/server/db/prisma.ts:2` uses `'../../src/generated/prisma/client'`, which is the canonical re-export entry (re-exports everything from `./enums.js`). Adopted the same path for consistency. The plan's grep acceptance pattern (`from '../../src/generated/prisma'`) still matches via substring.
- **Did NOT tighten subscriptionService.ts:152/186 (optional):** PATTERNS.md and the plan's Step 2 explicitly leave this as the executor's call. Skipped to keep the commit single-file and minimize cross-plan blast radius.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking] Stale root prisma.config.ts pointed to a duplicate schema**

- **Found during:** Task 1 (db push)
- **Issue:** First `npx prisma db push --accept-data-loss --config=../../prisma.config.ts` silently loaded `D:\NewsHub\prisma\schema.prisma` (a stale duplicate at the repo root, 517 lines) instead of the canonical `apps/web/prisma/schema.prisma` (587 lines, the file Plan 01 modified). The root duplicate lacks the `ApiKey` model added in Phase 35, so the first push **dropped the ApiKey table** (1 row of test data) along with successfully creating Campaign, ProcessedWebhookEvent, ReferralReward, StudentVerification, and the two new enums. Caught immediately by inspecting `\dt` output and seeing `ApiKey` missing.
- **Fix:** Re-ran `npx prisma db push --accept-data-loss --config=../../prisma.config.ts --schema=prisma/schema.prisma` from `apps/web` cwd. The explicit `--schema` flag overrides the config's `schema:` field and points at the canonical schema. Result: ApiKey table re-created, all 4 enum types present, 25 tables total. Per D-02 the dev DB is disposable; the lost ApiKey row was test data only (no real users have keys yet — Phase 35.1 hotfix verified by UAT 5/5 PASS, no production data).
- **Files modified:** Live dev DB only (no source code changes for this fix; the workaround is operational).
- **Verification:** Second db push reports "already in sync" (idempotent). `\dt` shows all 25 tables including ApiKey. `\dT+` shows all 4 enum types with correct value lists.
- **Committed in:** N/A (DB-only operation; the workaround is documented in commit `556694f` body)
- **Long-term recommendation (out of scope for 36.2-03):** Either delete `D:\NewsHub\prisma\schema.prisma` outright (it is a stale leftover from before the monorepo migration), or fix `D:\NewsHub\prisma.config.ts` to point at `apps/web/prisma/schema.prisma`. Tracked for a future tidy-up plan.
