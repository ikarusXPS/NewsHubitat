## Accomplishments

- Added 4 new Prisma models in a `// PHASE 36.2: STRIPE WEBHOOK + REFERRALS + STUDENT DISCOUNT + CAMPAIGNS` section placed before `model ApiKey`:
  - `ProcessedWebhookEvent` — webhook idempotency record with caller-supplied Stripe event ID as natural key (per D-08); single `@@index([processed])` for cleanup-job queries.
  - `ReferralReward` — two named relations (`ReferralRewardReferrer`, `ReferralRewardReferred`) to User, both `onDelete: Cascade`; `@@index` on both FKs (per D-04).

- Added 8 new User fields organized into 4 commented sub-blocks: Stripe (subscriptionEndsAt + pausedUntil), Premium perks (showPremiumBadge, customAccentColor), Referrals (referralCode @unique, referredBy, freeMonthsEarned), Student discount (isStudent, studentVerifiedUntil).
- Added 5 new User back-relations: 2 for ReferralReward (named), the UserReferrer self-relation pair (referredByUser + referredUsers, with `onDelete: SetNull` per D-06), and studentVerifications.

## Decisions Made

All decisions follow the locked D-01..D-08 rulings from `36.2-CONTEXT.md`:

- **D-01 (UPPERCASE enums)** — Picked over lowercase to match existing string literals in `subscriptionService.ts` (`?? 'FREE' as SubscriptionTier`) and 26 test fixtures, yielding zero call-site churn. Accepts known inconsistency with `enum ApiKeyTier { free pro }` (out of scope to harmonize).

I honored this even though `prisma validate` fails at the Task 1 checkpoint, because the plan explicitly states `Do NOT yet edit the User model field block in this task — that is Task 2.` The schema-text-only history is more reviewable than a single 109-line monolithic commit.
