<decisions>
## Implementation Decisions

### Enum Migration

- **D-01:** Use **UPPERCASE** enum values: `enum SubscriptionTier { FREE PREMIUM ENTERPRISE }` and `enum SubscriptionStatus { ACTIVE PAST_DUE CANCELED PAUSED }`. Rationale: matches the literal string values currently stored in the `String?` columns and the spec in `36-CONTEXT.md`. Lowest churn for `subscriptionService.ts` (~10 consumer sites) and tests. Inconsistency with existing `enum ApiKeyTier { free pro }` is accepted as known and not in scope to harmonize.
- **D-02:** Apply via `npx prisma db push --accept-data-loss` against the dev DB. Same approach 36.1 used (commit `a0df872`). Dev DB is treated as disposable; the 5 existing `subscriptionTier`/`Status` columns currently hold `NULL` for all users (no real subscription data; PAY-01 ships in 36-05 E2E tests only). The plan must include a brief `PRODUCTION-MIGRATION.md` note in the phase directory documenting that production cutover requires a manual `ALTER COLUMN ... TYPE ... USING` SQL migration (not in scope for 36.2 — flagged for the production-readiness phase).
- **D-03:** Replace the TS unions in `apps/web/server/config/stripe.ts` with re-exports from the generated Prisma client: `export type { SubscriptionTier, SubscriptionStatus } from '../../src/generated/prisma'`. Single source of truth. Because D-01 picks UPPERCASE that matches the existing string literals, **zero call-site changes are needed** — `?? 'FREE' as SubscriptionTier` keeps working, all test fixtures keep their literal strings, and `PRICE_TO_TIER` keeps its UPPERCASE keys. The implicit Status union (currently spelled inline as `'ACTIVE' | 'PAST_DUE' | 'CANCELED' | 'PAUSED'`) is replaced by the imported enum.

### New Model Relations

- **D-04:** `ReferralReward.referrerId` and `ReferralReward.referredId` are **both Prisma relations to `User` with `onDelete: Cascade`**. Two named relations on `User`: `referralRewardsAsReferrer ReferralReward[] @relation("ReferralRewardReferrer")` and `referralRewardsAsReferred ReferralReward[] @relation("ReferralRewardReferred")`. Indexes on both `referrerId` and `referredId`.
- **D-05:** `StudentVerification.userId` is a **Prisma relation to `User` with `onDelete: Cascade`**. Adds `studentVerifications StudentVerification[]` on the `User` model. Indexes on `userId` and `status` (matches the spec from `36-CONTEXT.md`).
- **D-06:** `User.referredBy` is the **`User.id` of the referrer** as a self-relation, not a code string. Schema:
  ```prisma
  referredBy   String?
  referredByUser User?  @relation("UserReferrer", fields: [referredBy], references: [id], onDelete: SetNull)
  referredUsers  User[] @relation("UserReferrer")
  ```
  `referralCode` stays as a separate `String? @unique` field on User (the *referrer's* code that *other users* enter at signup). At signup the lookup pattern is: `findUnique({ where: { referralCode } })` to resolve the referrer, then store `referredBy = referrer.id`. `onDelete: SetNull` on the relation so deleting a referrer doesn't cascade-delete the people they referred.
- **D-07:** `Campaign` has **no FK relations** in 36.2 (matches the spec from `36-CONTEXT.md`). Future phases that introduce per-campaign usage tracking will add a `CampaignUsage` join model rather than direct FKs from `Campaign` to `User`.
- **D-08:** `ProcessedWebhookEvent` matches the orphaned root code's expectations exactly (`prisma.processedWebhookEvent.findUnique({ where: { id: eventId } })` and `.create({ data: { id, eventType } })`): `id String @id`, `eventType String`, `processed DateTime @default(now())`, `@@index([processed])`. No relation needed. The 24h Redis idempotency layer is the primary; the DB row is the durable backup. Cleanup of old rows is deferred to a future phase (existing `cleanupService.ts` pattern can be extended later).

### Depcheck Cleanup Verification Gate

- **D-09:** Cleanup is gated by **all five** verification steps in this exact order: (1) edit `apps/web/package.json` to remove the 5 deps, (2) `pnpm install --no-frozen-lockfile` from the repo root to regenerate `pnpm-lock.yaml`, (3) re-run depcheck against `apps/web` and confirm no new transitive offenders appear in the output, (4) `pnpm typecheck` exits 0 across all workspaces, (5) `pnpm test:run` exits 0 with no regressions. All five must pass before the cleanup task is committed.
- **D-10:** Schema work and depcheck cleanup live in **separate plans** with distinct commits, so a regression in one does not block the other from being shipped. Plan ordering: schema models first (since 36.3 unblocks on `ProcessedWebhookEvent`), then the enum migration, then depcheck cleanup, then the audit annotation. Allows the planner to assign the schema/enum work to wave 1 and the cleanup + annotation to wave 2.

### Audit Annotation Format

- **D-11:** Append a new `## Phase 36 Debt Closure (Audit Trail)` section to the **bottom** of `36-01-SUMMARY.md`. Each row in the existing "Known Stubs" table gets one line in the new section citing closing phase + commit. Format: `- **{Item}** — closed by Phase {N} ({short SHA} | {commit subject})`. The original `## Known Stubs` table is left untouched (history preservation). The annotation also notes which items remain `MISSING` (none, after 36.2 — but the format must support partial closure for future debt cycles).

## Existing Code Insights

### Reusable Assets
- **Existing enum convention** — `enum ApiKeyTier { free pro }` and `enum ApiKeyEnv { live test }` in `schema.prisma` (lines 477–485) demonstrate how Prisma enums are written in this project. 36.2 will write the new enums in the same file but with UPPERCASE values per D-01.
- **Existing FK + cascade pattern** — `TeamMember.user User @relation(fields: [userId], references: [id], onDelete: Cascade)` (line 403) is the analog for `StudentVerification.userId` (D-05) and `ReferralReward.referrerId/referredId` (D-04).
- **Existing self-relation pattern** — `Comment.parentComment Comment? @relation("Replies", fields: [parentId], references: [id], onDelete: NoAction)` (line 361) is the analog for the `User.referredByUser` self-relation in D-06 (with `onDelete: SetNull` instead of `NoAction`).
- **Existing `?? fallback` pattern** — `subscriptionService.ts:151` (`?? 'FREE' as SubscriptionTier`) is the established read-path idiom; D-01's UPPERCASE choice keeps this idiom working.

### Established Patterns
- **`prisma db push` over `prisma migrate dev`** — set in 36.1 (a0df872). The `apps/web/prisma/migrations/` folder is sqlite-locked from a long-abandoned init and is **not** the live migration source. Plan must use `db push --accept-data-loss` per D-02.
- **Field-level `@unique` over `@@unique([...])`** — set in 36.1; matches `googleIdHash @unique` and `stripeCustomerId @unique` in the live schema. `User.referralCode` in 36.2 follows the same pattern.
- **Soft-typing at the service boundary** — Prisma generates the canonical types; service code re-exports or aliases them rather than re-declaring. D-03 makes this explicit for `SubscriptionTier`/`SubscriptionStatus`.

### Integration Points
- **Generated Prisma client** — `apps/web/src/generated/prisma/` must be regenerated after every `prisma db push`. The plan must include `npx prisma generate` as an explicit task. Without regeneration, the new models/fields/enums are not visible to TypeScript consumers.
- **`packages/types`** — currently does not re-export Prisma types. 36.2 does not change this (out of scope); `SubscriptionTier`/`SubscriptionStatus` re-export lives in `apps/web/server/config/stripe.ts` (single consumer surface).
- **Phase 36.3 dependency** — `ProcessedWebhookEvent` must be present in the generated Prisma client before 36.3 can wire `apps/web/server/services/stripeWebhookService.ts`. 36.2 → 36.3 sequencing is enforced via the `Depends on:` line in ROADMAP.md.
