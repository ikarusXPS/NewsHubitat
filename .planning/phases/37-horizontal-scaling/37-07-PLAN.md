# Multi-Region Deployment Patterns

> Phase 37 / INFRA-05 deliverable. Architecture-only — no vendor templates.
> Implementation lives in INFRA-F01 (active-active), INFRA-F02 (Kubernetes), INFRA-F03 (CDN edge).
> Last updated: 2026-04-29

This document describes patterns for deploying NewsHub across multiple regions
when single-region capacity (currently sized for ~30k concurrent users via Phase
37's Swarm topology) is no longer sufficient. It does NOT contain deployment
recipes for any specific cloud vendor — vendor-specific guidance belongs in
INFRA-F01 territory, where the actual implementation phase will pick a target
provider and build the topology.

## 1. When to go multi-region

Multi-region deployment is justified by ONE of:

- **Latency goals.** If users in Asia routinely see > 200ms RTT to your EU/US
  primary region and that hurts engagement, geographic distribution can pull p95
  global latency below 100ms. Realistic target: < 100ms p95 from each populated
  region to the nearest replica.
- **Data residency obligations.** GDPR (EU-resident user data must stay in EU),
  PIPL (China), LGPD (Brazil), and several other regulations require user data
  to be stored / processed in specific jurisdictions. NewsHub does not currently
  trigger any of these for its current user base, but expansion plans should
  trigger a residency analysis.
- **Availability targets.** A single-region deployment has a hard RTO floor at
  ~minutes (the time to spin up a fresh region from backups). RPO depends on
  backup cadence — typically minutes to an hour. If contractual or product
  commitments require RTO < 5 min and RPO < 1 min, multi-region active-passive
  (or active-active) becomes mandatory.

What 30k concurrent users does NOT need: multi-region is a constant operational
tax (deploy ordering, replication lag monitoring, conflict-resolution runbooks).
At the current load profile, the marginal cost outweighs the marginal benefit.

What 300k+ concurrent users probably DOES need: at this scale, single-region
capacity headroom shrinks to the point where regional outages become
material — and the global latency profile starts to matter for retention.

## 2. Active-active vs active-passive

**Active-passive** (also "primary-secondary"): one region serves all traffic.
The other regions hold near-real-time replicas of state, ready to take over on
failover. Lower operational complexity (no conflict resolution); higher RTO
(failover takes seconds to minutes). Good for HA-driven multi-region.

**Active-active**: all regions serve traffic simultaneously, each region's reads
go to its local replicas, and writes are either:
- Pinned to a single home region per user (per-user routing) — avoids most
  conflicts but loses the latency benefit for write-heavy users
- Routed to nearest region (any region can write) — requires conflict
  resolution, typically Last-Write-Wins (LWW) or CRDTs

Active-active fits NewsHub's read-heavy profile reasonably well: most user
interactions are reads (browse articles, view perspective comparisons), and the
write path (Comments, Bookmarks, Team interactions) tolerates eventual
consistency well as long as read-after-write within a region is preserved.

The single hard write path is RSS aggregation by `app-worker` (Phase 37,
JOB-01). That worker MUST remain a global singleton — it should run in exactly
one region (typically primary or by leader election) to prevent duplicate writes
across regions. Cross-region replicas of NewsArticle propagate via Postgres
replication.

## 3. Postgres replication options

| Option | When it fits | Trade-off |
|--------|--------------|-----------|
| **Streaming replication** (built-in) | Single-primary; reads can fan out to local replicas | Async lag (typically < 1 sec); failover is manual or via patroni/repmgr |
| **Logical replication** (built-in) | Selective table sync; supports cross-version upgrades; bidirectional setups | More moving parts; conflict handling is your problem in bidirectional mode |
| **Managed: Aurora Global Database** (AWS) | Multi-region read replicas with sub-second cross-region lag | AWS-specific, paid; failover is well-tooled |
| **Managed: Cloud SQL cross-region read replicas** (GCP) | Similar to Aurora Global, GCP-flavored | GCP-specific, paid |
| **Managed: Neon branching / replication** | Modern serverless Postgres; branching is useful for staging+production parity | Newer; vendor lock-in less than the hyperscalers but still real |

**Multi-master Postgres is hard.** Solutions exist (BDR, pglogical-based
multi-master, Yugabyte) but conflict resolution becomes application logic. For
NewsHub specifically, the read-heavy profile means a single primary + per-region
read replicas captures most of the latency win without the conflict-resolution
tax. Recommendation for INFRA-F01: start with single-primary streaming
replication; revisit multi-master only if write latency becomes a measured
problem (it currently isn't).

## 4. Redis replication options

| Option | When it fits | Trade-off |
|--------|--------------|-----------|
| **Master-replica** (built-in) | Async replication; eventually consistent | Replicas are read-only; failover requires Sentinel or external orchestration |
| **Redis Cluster** (built-in) | Sharded; each shard has master + replicas; horizontally scalable | Operational complexity grows; multi-key transactions across shards are forbidden |
| **Managed: Elasticache cross-region replication groups** | AWS-managed multi-region | AWS-specific, paid |
| **Managed: Upstash Global** | Globally-distributed Redis with strong eventually-consistent reads | Newer; cost model is per-operation, can be expensive at scale |

**Important Phase 37 note:** the `@socket.io/redis-adapter` (WS-01) currently
assumes a SINGLE Redis Pub/Sub plane. In a multi-region setup, that single
plane becomes a cross-region fanout bottleneck (publishes from region A pay
inter-region latency to reach subscribers in region B).

For multi-region WebSocket fanout, the recommended pattern is:
- Each region runs its own Redis instance and its own `redis-adapter`
- Cross-region fanout (when a write in region A should reach clients in
  region B) flows via Redis Streams replicated cross-region, with a
  per-region consumer that re-emits onto the local adapter

This is INFRA-F01 territory — Phase 37 does not implement it.

## 5. Latency-vs-consistency trade-offs

Framed by CAP: in a partition, you must choose Availability or Consistency.
NewsHub's data domains have different needs:

| Data domain | Consistency need | Multi-region pattern |
|-------------|------------------|----------------------|
| **NewsArticle (RSS-aggregated)** | Eventual consistency is fine. Articles are immutable once written; cross-region replication lag of a few seconds doesn't hurt UX. | Single-region writes (worker singleton); async replication to read regions |
| **NewsSource metadata** | Static, edited rarely. Eventual consistency is fine. | Same |
| **User accounts** | Read-after-write within a region is required (just-registered user must be able to log in). Cross-region read-after-write isn't required (the user only logs in from their home region). | Per-user pinning (route user to home region); replicate User table for failover |
| **Comments** | Read-after-write within a region is required (user posts a comment, must see it on refresh). Cross-region is eventually consistent. | Region-local writes; logical replication for cross-region read |
| **Bookmarks, ReadingHistory, Team bookmarks** | Same as Comments. | Same |
| **Subscription / Stripe state** | Strong consistency: a payment confirmation must be globally visible immediately. | Pin all subscription / Stripe webhook handling to the home region (single-primary). |
| **AI cache (Redis)** | Eventually consistent; cache misses re-compute. | Region-local Redis. |

NewsHub leans heavily eventually-consistent because most state is read-mostly
news data. The exceptions (Auth, Subscription) get single-region treatment.

## 6. CDN edge caching

A regional edge cache layer (Cloudflare, Fastly, AWS CloudFront, Bunny) reduces
the latency win that multi-region requires from "deploy in 4 regions" to
"deploy in 2 regions + edge cache". For NewsHub specifically:

**Cacheable at edge:**
- `GET /api/news` — already has `Cache-Control: public, max-age=300` (apps/web/server/routes/news.ts)
- `GET /api/news/sources` — static-ish; longer cache acceptable
- Static frontend assets — already cacheable

**NOT cacheable at edge:**
- `GET /api/auth/me` — per-user; would leak user identity if cached
- `GET /api/account/*` — per-user
- `GET /api/history` — per-user
- `POST /*` and `PUT /*` and `DELETE /*` — never cache mutations
- WebSocket connections — long-lived; CDN typically passes through unchanged

CDN edge caching is INFRA-F03 — Phase 37 does not implement, this doc only
points at the strategy.

## 7. Operational considerations

**DNS-based routing.** Multi-region deployment requires intelligent DNS:
- **GeoDNS** (route by client geographic IP) — simple, coarse-grained
- **Latency-based routing** (e.g. AWS Route 53 latency-based routing) — measures
  actual RTT, picks lowest-latency region per client
- **Anycast** (Cloudflare, Fastly) — single IP advertised from all regions; the
  network routes to nearest BGP peer

**Deploy ordering.** Multi-region rolling deploys must be sequenced:
- Deploy schema migrations to ALL regions FIRST (DIRECT_URL bypassing PgBouncer
  per Phase 37 / DB-02), or use blue-green: stand up new schema region, drain
  old region, swap routing
- App rollout: deploy to one region, monitor for 15 minutes, then proceed to
  the next region — minimizes blast radius

**Monitoring.** Prometheus federation (each region's Prometheus instance feeds
a global aggregator) is the standard pattern. Phase 20 already runs Prometheus;
multi-region adds federation rules.

**Incident response.** Per-region runbooks (which region is hot, what the
failover procedure is, who has paging access) are operational, not architectural.

## 8. What this phase does NOT do

- **Active-active multi-region implementation.** Tracked under INFRA-F01.
- **Kubernetes migration.** Phase 37 ships on Docker Swarm. Migration to
  Kubernetes (for richer multi-region orchestration) is INFRA-F02.
- **CDN edge caching implementation.** Tracked under INFRA-F03.
- **Vendor selection.** This doc deliberately does not name a primary cloud
  vendor — that decision belongs to INFRA-F01.
