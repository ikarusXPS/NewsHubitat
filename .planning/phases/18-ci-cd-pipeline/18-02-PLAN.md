**Job 6: deploy-staging** (per D-09, D-10, D-11)
- Runner: `ubuntu-latest`
- Condition: `if: github.ref == 'refs/heads/main'` (only deploy on main branch, not PRs)
- Depends on: `needs: [build, e2e]` (wait for both build and e2e to pass)
- Environment:
  ```yaml
  environment:
    name: staging
    url: https://staging.newshub.example.com
  ```
- Steps:
  1. Deploy via SSH using `appleboy/ssh-action@029f5b4aeeeb58fdfe1410a5d17f967dacf36262` (commit SHA per security best practice)
     - host: `${{ secrets.STAGING_HOST }}`
     - username: `${{ secrets.STAGING_USER }}`
     - key: `${{ secrets.STAGING_SSH_KEY }}`
     - script (D-09 pattern):
       ```bash
       cd /app/newshub
       docker compose pull
       docker compose up -d --wait
       docker compose ps
       curl -f http://localhost:3001/api/health/db || exit 1
       ```
  2. The `--wait` flag ensures containers are healthy before continuing
  3. Health check verification via curl ensures deployment succeeded

**Job 7: deploy-production** (per D-09, D-10, D-11)
- Runner: `ubuntu-latest`
- Condition: No `if` condition (runs after staging, but pauses for approval)
- Depends on: `needs: deploy-staging` (sequential deployment: staging first, then production)
- Environment:
  ```yaml
  environment:
    name: production
    url: https://newshub.example.com
  ```
- Steps: Same as staging, but using PRODUCTION_HOST, PRODUCTION_USER, PRODUCTION_SSH_KEY
- Approval gate: Configured in GitHub Environments UI (not in workflow file)

**Key differences from staging**:
- Production uses `environment: production` which has required reviewers configured via GitHub UI
- Workflow pauses at production deploy job until reviewer approves
- Same SSH deployment script (D-09: docker compose pull && up -d)

**Security notes** (per RESEARCH.md):
- Secrets are environment-scoped (staging secrets only available to staging job, production secrets only to production job)
- No repository-level secrets used (prevents accidental cross-environment access)
- SSH action pinned to commit SHA (not version tag)
- Health check verification ensures deployment succeeded before marking job complete

**Deployment flow**:
1. PR merged to main → workflow triggers
2. Quality gates run → build → e2e
3. deploy-staging job runs automatically (no approval)
4. deploy-production job pauses waiting for approval
5. Reviewer approves via GitHub UI
6. deploy-production job executes

Set up GitHub Environments for staging and production deployment approval gates.

**Why human-only:** GitHub Environments are configured via repository Settings UI, not via code or API.

**Actions required:**

1. **Create staging environment** (no protection):
   - Go to: Repository Settings → Environments → New environment
   - Name: `staging`
   - Protection rules: None (leave empty)
   - Environment secrets: Add 3 secrets
     - `STAGING_HOST`: SSH hostname (e.g., `staging.newshub.example.com` or IP address)
     - `STAGING_USER`: SSH username (e.g., `deploy`)
     - `STAGING_SSH_KEY`: Private SSH key (entire contents, including `-----BEGIN OPENSSH PRIVATE KEY-----`)
   - Environment URL: `https://staging.newshub.example.com` (optional, for deployment history)

2. **Create production environment** (with approval):
   - Go to: Repository Settings → Environments → New environment
   - Name: `production`
   - Protection rules:
     - ✓ Required reviewers: Select yourself or team members (at least 1)
     - Wait timer: 0 minutes (optional: add delay if needed)
   - Environment secrets: Add 3 secrets
     - `PRODUCTION_HOST`: SSH hostname (e.g., `newshub.example.com` or IP address)
     - `PRODUCTION_USER`: SSH username (e.g., `deploy`)
     - `PRODUCTION_SSH_KEY`: Private SSH key (entire contents)
   - Environment URL: `https://newshub.example.com` (optional)

3. **Verify SSH key format**:
   - Private key should include full contents:
     ```
     -----BEGIN OPENSSH PRIVATE KEY-----
     b3BlbnNzaC1rZXktdjEAAAAABG5vbmUAAAAEbm9uZQAAAAAAAAABAAABlwAAAAdzc2gtcn
     ... (many lines) ...
     -----END OPENSSH PRIVATE KEY-----
     ```
   - If key has passphrase, remove it: `ssh-keygen -p -f ~/.ssh/deploy_key` (set empty passphrase)
   - Public key should be added to server's `~/.ssh/authorized_keys`

4. **Test SSH connectivity** (optional but recommended):
   ```bash
   ssh -i ~/.ssh/deploy_key deploy@staging.newshub.example.com "docker compose version"
   ```
   - Should connect without password prompt
   - Should return docker compose version

**Server requirements:**
- Docker and docker compose installed on both staging and production servers
- NewsHub code repository cloned to `/app/newshub` on both servers
- `.env` file configured on both servers with environment-specific values
- SSH key's public key added to `~/.ssh/authorized_keys` for the deploy user

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions → Staging/Production servers | SSH connection with private key authentication |
| Staging environment → Production environment | Separate secrets prevent cross-contamination |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-06 | Information Disclosure | SSH private keys in logs | mitigate | GitHub auto-masks secrets in logs. SSH action never echoes key values. Use `${{ secrets.* }}` syntax only. |
| T-18-07 | Elevation of Privilege | Staging secrets used in production | mitigate | Environment-scoped secrets: staging job cannot access PRODUCTION_* secrets. GitHub enforces separation. |
| T-18-08 | Tampering | Unauthorized production deployment | mitigate | Required reviewers on production environment. Workflow pauses until approved. Audit trail in Environments page. |
| T-18-09 | Denial of Service | Failed deployment takes down production | mitigate | Health check verification (`curl -f /api/health/db || exit 1`). Deployment fails if health check fails. Docker compose --wait ensures containers are healthy. |
| T-18-10 | Information Disclosure | Production secrets exposed to PRs | mitigate | Deployment jobs have `if: github.ref == 'refs/heads/main'`. PRs never execute deployment jobs. |
</threat_model>
