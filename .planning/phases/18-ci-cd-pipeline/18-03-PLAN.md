## Phase 1: Pull Request Quality Gates (CICD-01)

1. **Create test branch and make a small change:**
   ```bash
   git checkout -b test-ci-pipeline
   # Make a trivial change (add comment to README.md or update a log message)
   git add .
   git commit -m "test: verify CI pipeline"
   git push origin test-ci-pipeline
   ```

2. **Create pull request:**
   - Go to GitHub repository
   - Create PR from `test-ci-pipeline` to `main`
   - Title: "Test: CI Pipeline Verification"

3. **Verify quality gates run:**
   - PR should show "Checks" section with workflow run
   - Click "Details" to view workflow
   - Verify 5 jobs appear: lint, typecheck, test, build, e2e
   - Verify lint, typecheck, test run in parallel (check start times)
   - Verify build waits for all three to pass
   - Verify e2e runs after build
   - **Expected time:** ~5-7 minutes total (parallel execution)

4. **Verify service containers:**
   - Expand "test" job logs
   - Verify PostgreSQL service started: "PostgreSQL 17 is healthy"
   - Verify Redis service started: "Redis 7.4 is healthy"
   - Verify Prisma migrations applied
   - Verify tests pass with coverage report

5. **Verify E2E tests:**
   - Expand "e2e" job logs
   - Verify Playwright browsers installed: "chromium installed"
   - Verify E2E tests run (should see test names in logs)
   - Verify all 62 tests pass (or current test count)

6. **Verify Docker build (no push on PR):**
   - Expand "build" job logs
   - Verify buildx cache hit/miss status
   - Verify image built successfully
   - Verify "push: false" (should NOT push to GHCR on PR)
   - **Expected:** Build completes but no "Pushing to ghcr.io" message

7. **Verify no deployment on PR:**
   - Workflow should show only 5 jobs (no deploy jobs)
   - deploy-staging and deploy-production should be skipped
   - PR checks section should show green checkmarks

## Phase 2: Docker Image Push to GHCR (CICD-02)

8. **Merge pull request:**
   - Approve and merge PR to main branch
   - Wait for workflow to trigger on main

9. **Verify Docker image push:**
   - Go to Actions tab → Find main branch workflow run
   - Expand "build" job logs
   - Verify "Pushing to ghcr.io" appears in logs
   - Verify tags pushed:
     - `ghcr.io/[owner]/newshub:[commit-sha]`
     - `ghcr.io/[owner]/newshub:latest`
     - `ghcr.io/[owner]/newshub:main`

10. **Verify GHCR.io package:**
    - Go to repository main page
    - Click "Packages" in right sidebar (or go to Packages tab)
    - Click "newshub" package
    - Verify tags appear: latest, main, [commit-sha]
    - Verify package size and creation time
    - **Expected:** Package shows "pushed X minutes ago"

## Phase 3: Staging Deployment (CICD-03)

11. **Verify staging deployment job runs:**
    - In main branch workflow run, verify deploy-staging job appears
    - Job should run automatically (no approval needed)
    - Expand job logs
    - Verify SSH connection successful
    - Verify deployment commands executed:
      ```
      cd /app/newshub
      docker compose pull
      docker compose up -d --wait
      docker compose ps
      curl -f http://localhost:3001/api/health/db
      ```
    - Verify health check passes (exits 0)

12. **Verify staging server updated:**
    - If you have access to staging server, SSH in and verify:
      ```bash
      ssh deploy@staging.newshub.example.com
      cd /app/newshub
      docker compose ps  # Should show running containers
      docker compose logs app --tail 20  # Should show recent startup logs
      ```
    - Verify application responds: `curl http://localhost:3001/api/health/db`
    - **Expected:** `{"status":"healthy","database":"connected"}`

13. **Verify GitHub Environments page:**
    - Go to repository → Environments → staging
    - Verify deployment appears in history
    - Verify deployment status: "Active"
    - Verify deployment time and commit SHA
    - **Expected:** Most recent deployment matches main branch commit

## Phase 4: Production Approval Gate (CICD-04)

14. **Verify production deployment pauses:**
    - In main branch workflow run, verify deploy-production job appears
    - Job should show orange status: "Waiting"
    - Hover over job or click to see: "Waiting for approval"
    - Verify yellow banner: "This workflow is waiting for approval from required reviewers"

15. **Verify approval UI:**
    - Click "Review deployments" button
    - Verify production environment listed
    - Verify your username as required reviewer
    - **Do NOT approve yet** (verify pause works)
    - Close modal

16. **Verify workflow stays paused:**
    - Refresh page after 1-2 minutes
    - Verify deploy-production still shows "Waiting" status
    - Verify other jobs completed (not blocking)
    - **Expected:** Production job can wait indefinitely for approval

17. **Approve production deployment:**
    - Click "Review deployments" again
    - Check "production" environment
    - Optional: Add comment "Approved for testing"
    - Click "Approve and deploy" button

18. **Verify production deployment executes:**
    - deploy-production job should change from "Waiting" to "Running"
    - Expand job logs
    - Verify SSH connection to production server
    - Verify same deployment commands as staging
    - Verify health check passes
    - Job should complete with green checkmark

19. **Verify production server updated:**
    - SSH to production server (if accessible):
      ```bash
      ssh deploy@newshub.example.com
      cd /app/newshub
      docker compose ps
      curl http://localhost:3001/api/health/db
      ```
    - Verify production environment variables are used (not staging)
    - **Expected:** Production database connected, not staging database

20. **Verify GitHub Environments audit trail:**
    - Go to Environments → production
    - Verify deployment history shows:
      - Deployment commit SHA
      - Deployment time
      - Approver username
      - Deployment status: "Active"
    - Click deployment to see full details and approval comment
    - **Expected:** Audit trail shows who approved and when

## Phase 5: Status Visibility (All Requirements)

21. **Verify PR status checks integration:**
    - Create another test PR
    - Go to PR page
    - Verify "Checks" section shows workflow status
    - Verify individual job status visible
    - Verify "Required" label on required checks
    - Merge button should be blocked until checks pass
    - **Expected:** Green "All checks have passed" when complete

22. **Verify commit status badges:**
    - Go to main branch commit history
    - Click on a recent commit
    - Verify green checkmark or red X next to commit SHA
    - Hover over status icon
    - Verify dropdown shows individual job statuses
    - **Expected:** Status visible directly in commit list

23. **Verify deployment status in PR:**
    - In merged PR view
    - Scroll to bottom timeline
    - Verify deployment events appear:
      - "Deployed to staging" (automatic)
      - "Deployed to production" (with approval)
    - Click deployment to go to Environments page
    - **Expected:** Deployment history linked from PR

## Phase 6: Cache Performance Validation

24. **Verify build cache effectiveness:**
    - Trigger another workflow run (push small change to main)
    - Expand build job logs
    - Look for cache hit messages:
      - "CACHED" status on Docker layers
      - "cache-hit=true" in npm cache restore
    - Compare build time to first run
    - **First run:** ~5 minutes for Docker build
    - **Cached run:** ~30 seconds for Docker build
    - **Expected:** 60-80% faster builds on cache hit

25. **Verify npm cache:**
    - In any job with `setup-node` (lint, typecheck, test, e2e)
    - Look for: "Cache restored from key: ..."
    - Verify `npm ci` completes in <30 seconds (vs 2+ minutes uncached)
    - **Expected:** Significant speedup on dependency installation

## Failure Scenarios (Optional but Recommended)

26. **Test failing quality gate:**
    - Create branch with intentional lint error
    - Push and create PR
    - Verify lint job fails with red X
    - Verify build job is skipped (not run)
    - Verify "Some checks were not successful" in PR
    - **Expected:** Failed quality gate blocks merge

27. **Test failing health check:**
    - Temporarily break `/api/health/db` endpoint on staging server
    - Trigger deployment
    - Verify deployment job fails at health check step
    - Verify deployment marked as failed in Environments page
    - **Expected:** Deployment fails if application not healthy
