<design_decisions>
## Critical CSS Deferral (per D-09 discretion)

D-07 specifies "Inline critical CSS for Dashboard shell (above-fold layout)" and D-09 grants Claude discretion on critical CSS tool selection.

**Decision:** Critical CSS extraction is deferred for this phase.

**Rationale:** The Dashboard shell pattern implemented in Task 1 uses skeleton animations (`animate-pulse`) that render immediately with the initial CSS bundle. This achieves the same user-perceived effect as critical CSS inlining:
- Above-fold layout structure is visible immediately
- Skeleton placeholders match approximate dimensions of real content
- No layout shift when content loads

The skeleton pattern provides:
1. Instant visual feedback (same as critical CSS)
2. Clear loading state affordance (better than blank space)
3. Zero build complexity (no critters/vite-plugin-critical configuration)

RESEARCH.md Pattern 3 (lines 281-317) demonstrates this skeleton approach as the standard Dashboard loading pattern. Adding critical CSS extraction would provide marginal benefit while adding build-time complexity and potential maintenance burden.

If future Lighthouse audits show FCP/LCP issues specifically attributable to CSS loading, critical CSS can be revisited in a dedicated optimization phase.
</design_decisions>

2. **Add bundle size check step AFTER the build step** but BEFORE any deploy steps:
```yaml
      # Bundle size warning (per D-16 - warn but don't block)
      - name: Check bundle size
        id: bundle-size
        run: |
          # Extract main chunk sizes from build output
          echo "## Bundle Size Report" >> $GITHUB_STEP_SUMMARY
          echo "" >> $GITHUB_STEP_SUMMARY

          # Find the main index chunk (not vendor chunks)
          MAIN_CHUNK=$(find dist/assets -name "index-*.js" -type f | head -1)
          if [ -n "$MAIN_CHUNK" ]; then
            SIZE=$(stat -f%z "$MAIN_CHUNK" 2>/dev/null || stat --printf="%s" "$MAIN_CHUNK" 2>/dev/null)
            SIZE_KB=$((SIZE / 1024))
            echo "Main chunk: ${SIZE_KB}KB" >> $GITHUB_STEP_SUMMARY

            # Warn if > 250KB (per D-16)
            if [ "$SIZE_KB" -gt 250 ]; then
              echo "" >> $GITHUB_STEP_SUMMARY
              echo "**Warning:** Main chunk exceeds 250KB target (${SIZE_KB}KB)" >> $GITHUB_STEP_SUMMARY
              echo "::warning::Bundle size ${SIZE_KB}KB exceeds 250KB target"
            else
              echo "" >> $GITHUB_STEP_SUMMARY
              echo "Bundle size within target" >> $GITHUB_STEP_SUMMARY
            fi
          fi

          # Also report total JS size
          TOTAL_JS=$(find dist/assets -name "*.js" -type f -exec stat -f%z {} + 2>/dev/null | awk '{s+=$1}END{print s}' || find dist/assets -name "*.js" -type f -exec stat --printf="%s\n" {} + 2>/dev/null | awk '{s+=$1}END{print s}')
          TOTAL_JS_KB=$((TOTAL_JS / 1024))
          echo "Total JS: ${TOTAL_JS_KB}KB" >> $GITHUB_STEP_SUMMARY

          # Link to bundle analysis
          echo "" >> $GITHUB_STEP_SUMMARY
          echo "See [bundle analysis artifact](../artifacts) for details" >> $GITHUB_STEP_SUMMARY
```

Note: The step uses `::warning::` to create a GitHub Actions warning annotation per D-16. It does NOT use `exit 1` so it won't block PRs per D-17.

3. **Platform compatibility**: The script uses both BSD stat (macOS) and GNU stat (Linux) syntax with fallback. GitHub Actions runners typically use Linux.
