## Interaction Patterns

### OAuth Popup Flow (D-08)

1. User clicks OAuth button
2. Popup opens (500x600, centered) via `useOAuthPopup` hook
3. User authenticates with provider
4. Callback page receives token, posts message to parent
5. Parent window receives message, updates AuthContext
6. Popup closes automatically

**Fallback:** If popup blocked, redirect to provider with state parameter for return

### Account Linking Security (D-01, D-02)

| Scenario | Behavior |
|----------|----------|
| New email (no existing user) | Create account, mark emailVerified=true (D-04) |
| Matching email, unverified account | Auto-verify and link (Claude's Discretion) |
| Matching email, verified account | Show ReAuthModal, require password |
| Different email than account | Block linking with error (D-02) |

### Unlink Protection (D-12)

| hasPassword | Other OAuth linked | Unlink allowed |
|-------------|-------------------|----------------|
| true | any | Yes |
| false | yes | Yes |
| false | no | No (show "Set password first" message) |

## Error States

| Error | Display | Recovery |
|-------|---------|----------|
| OAuth popup blocked | Toast: "Popup was blocked. Please allow popups and try again." | User allows popups |
| OAuth provider error | Toast: "Sign in failed. Please try again or use email." | User retries or uses email |
| Account linking failed (wrong password) | Inline error in ReAuthModal: "Incorrect password" | User re-enters password |
| Email mismatch | Toast: "Cannot link. OAuth email does not match your account email." | User uses correct account |
| Network error | Toast: "Connection failed. Please check your internet and try again." | User retries |
