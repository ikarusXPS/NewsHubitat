## Summary

Phase 36.3 is a relocation + cleanup phase, not a feature-build phase. Three source files (webhook service, webhook route, subscription routes) were written to the wrong directory during Phase 36-02. The live backend has never seen these files. The fix is `git mv` + two diff-edits to `apps/web/server/index.ts` + wholesale deletion of the orphan root trees.

Research confirmed all CONTEXT.md locked decisions hold against the live code. However, four landmines were found that CONTEXT.md does not address and the planner must handle: (1) the `apps/web/server/routes/webhooks/` directory does not exist — `git mv` will fail without `mkdir -p` first; (2) `authMiddleware` is already applied per-route in `subscriptions.ts` — adding it again at mount level per D-05 causes double DB + Redis lookups per request; (3) the root `tsconfig.json`/`tsconfig.app.json`/`tsconfig.node.json` trio is not in the D-07 deletion list but references orphan files being deleted; and (4) `STRIPE_WEBHOOK_SECRET` in `apps/web/.env` has an inline comment that `dotenv` strips correctly — but the value must be verified clean before D-09.

**Primary recommendation:** Add `mkdir -p apps/web/server/routes/webhooks` as a required pre-step before `git mv` of the webhook route. On authMiddleware double-application: the planner should document the known redundancy — do NOT edit `subscriptions.ts` source (violates D-02), instead let D-05 mount add the outer guard for defense-in-depth clarity, accepting two auth checks per subscription request.

## Landmines and Edge Cases

### L-01: `apps/web/server/routes/webhooks/` does not exist — `git mv` will fail
**Verified:** `ls D:/NewsHub/apps/web/server/routes/webhooks/` returns NOT FOUND. [VERIFIED: ls]

### L-02: `authMiddleware` is already applied per-route in `subscriptions.ts`
**Verified:** Lines 49, 112, 153 of `server/routes/subscriptions.ts` each call `authMiddleware` inline. [VERIFIED: grep]

D-05 locks the mount as `app.use('/api/subscriptions', authMiddleware, subscriptionRoutes)`. This results in `authMiddleware` running **twice** per request: once at mount, once at the matched route handler. Each invocation does: JWT verify (cheap) + Redis blacklist lookup + `prisma.user.findUnique`. Two DB queries per subscription request.

**D-02 forbids editing subscriptions.ts during relocation.** So the planner must document this as a known redundancy. It is not a correctness issue — both checks pass for valid tokens. The outer mount-level guard also provides clear intent. Flag this as technical debt for a future cleanup pass.

**Note:** The adjacent `/api/teams` route uses the same pattern — per-route `authMiddleware` only, no mount-level guard. D-05 is intentionally different for defense-in-depth.
