# Phase 36.3: Code Review Report

**Reviewed:** 2026-04-28T13:12:45Z
**Depth:** standard
**Files Reviewed:** 5
**Status:** issues_found

## Summary

Phase 36.3 mounts previously-orphaned Stripe webhook + subscription routes into the live `apps/web/server/index.ts` and re-creates a workspace-local `prisma.config.ts`. The structural relocation is correct, the raw-body invariant for HMAC verification is preserved, and idempotency uses the documented Redis+DB dual-storage pattern.

However, the relocation imported pre-existing defects that now become reachable in production for the first time. Three categories of BLOCKER concerns:

1. **Idempotency is not concurrency-safe** — the check-then-act pattern in `processWebhookIdempotently` allows handler double-execution under parallel webhook deliveries, which Stripe explicitly does (at-least-once delivery, retries on transient failures). Subscription state mutations are not idempotent at the application layer (e.g., paid-period extension, downgrade), so duplicate execution can corrupt billing state.
2. **All processing errors are swallowed with HTTP 200** in the webhook route, with a comment claiming the event is recorded in the idempotency store — but the most common failure mode (DB write rejection inside `processWebhookIdempotently` itself) means the event is NOT recorded, Stripe will not retry (because we returned 200), and the subscription mutation is silently lost.
3. **Internal error messages leak to clients** in `subscriptions.ts` (DB errors, Stripe SDK errors) — explicit violation of the project's security checklist ("Error messages don't leak sensitive data").

Additional warnings include dead-code branches from over-eager defaulting after Zod validation, missing fail-fast on unset `DATABASE_URL`, fragile `instanceof`-style string matching for Stripe error categorization, and inconsistent missing-user logging across webhook handlers.

The three relocated files (`stripe.ts`, `subscriptions.ts`, `stripeWebhookService.ts`) were moved byte-identical per Plan 02 D-02, so most defects predate Phase 36.3. They are still in scope because Phase 36.3 is the commit that activates them in the live server.

## Critical Issues

### CR-01: Webhook idempotency check is racy — handler can execute twice for the same event

**File:** `apps/web/server/services/stripeWebhookService.ts:18-57`
**Issue:** `processWebhookIdempotently` performs a check-then-act sequence with no locking:

1. Read Redis (line 27)
2. Read DB (line 34)
3. Run `handler()` (line 45)
4. Write Redis + DB (lines 48-53)

Stripe explicitly delivers webhooks at-least-once and retries on timeout / 5xx. If two deliveries of the same `event.id` arrive within the window between step 2 and step 4 (e.g., a few hundred milliseconds during a slow `subscriptionService.getStripeSubscription()` call on line 121), both pass the existence checks, both call `handler()`, and only the second `prisma.processedWebhookEvent.create()` fails with `P2002` (unique-PK violation). At that point the side effects in `handler()` have already run twice.

Concrete consequences:
- `handleInvoicePaid` (line 183) writes `subscriptionEndsAt: new Date(invoice.period_end * 1000)` — same value, idempotent. OK.
- `handleSubscriptionDeleted` (line 165) calls `downgradeToFree`, which sets `stripeSubscriptionId: null`. First call wins, second call is a no-op. OK.
- `handleCheckoutCompleted` (line 104) calls `updateUserSubscription`, which writes `stripeCustomerId`, `stripeSubscriptionId`, tier, status. Same data, idempotent. OK in steady state.
- BUT: any future handler that does counter-style mutation (e.g., "increment quota on renewal", "send welcome email") will fire twice. The `// TODO: Send welcome email` comment on line 131 of `handleCheckoutCompleted` is the canonical example — once that TODO is filled in, races send the user two welcome emails per checkout.

The `P2002` unique-constraint violation also propagates up out of `processWebhookIdempotently`, gets caught by the route's blanket `catch` (stripe.ts:71), and returns 200 — see CR-02 below for why this is also a problem.

**Fix:** Use the unique constraint as the lock by attempting `create()` first inside a try/catch, and only run `handler()` on successful create:

```ts
export async function processWebhookIdempotently(
  eventId: string,
  eventType: string,
  handler: () => Promise<void>
): Promise<{ processed: boolean; duplicate: boolean }> {
  const cacheService = CacheService.getInstance();
  const cacheKey = `webhook:stripe:${eventId}`;

  // Fast path: Redis hit means definitely processed
  const cached = await cacheService.get<boolean>(cacheKey);
  if (cached) return { processed: false, duplicate: true };

  // Acquire the lock by inserting the row. Unique-PK enforces
  // mutual exclusion across concurrent deliveries.
  try {
    await prisma.processedWebhookEvent.create({
      data: { id: eventId, eventType },
    });
  } catch (err) {
    // P2002 = unique constraint violation = another process already
    // claimed this event. Refresh the cache and return duplicate.
    if ((err as { code?: string }).code === 'P2002') {
      await cacheService.set(cacheKey, true, IDEMPOTENCY_TTL);
      return { processed: false, duplicate: true };
    }
    throw err;
  }

  // We hold the lock. Run the handler.
  try {
    await handler();
    await cacheService.set(cacheKey, true, IDEMPOTENCY_TTL);
    return { processed: true, duplicate: false };
  } catch (err) {
    // Handler failed AFTER we claimed the lock. Release the row so
    // Stripe's retry can re-process. (Otherwise the event is poisoned
    // and never replays.)
    await prisma.processedWebhookEvent.delete({ where: { id: eventId } })
      .catch(() => {/* best effort */});
    throw err;
  }
}
```

The "delete on handler failure" branch is what makes CR-02 fixable: now a handler exception genuinely means "not yet recorded", so the route can safely return 5xx and let Stripe retry.

### CR-02: Webhook route silently swallows ALL processing errors with HTTP 200, dropping events on the floor

**File:** `apps/web/server/routes/webhooks/stripe.ts:67-76`
**Issue:** The route's outer try/catch returns `200 { received: true, warning: 'Processing error logged' }` for every error in `handleStripeWebhook`. The justifying comment says:

```
// Return 200 to prevent Stripe retries for processing errors
// (we've already recorded the event in idempotency store)
```

This claim is false in the most important failure modes. `handleStripeWebhook` calls `processWebhookIdempotently`, which can fail BEFORE recording the event:

- Redis fetch error (network blip) — caught internally by `cacheService.get`, returns null. OK.
- DB read error on `findUnique` (line 34) — bubbles up. **Event not recorded, returns 200, Stripe never retries → state mutation lost.**
- Handler exception (e.g., `prisma.user.update` rejected because the user was deleted between Stripe's send and our process) — bubbles up. **Event not recorded, returns 200, Stripe never retries → user stays on stale subscription state.**
- `Promise.all` on line 48 rejects on either Redis OR DB write — **handler ran but record may not be persisted; returns 200; Stripe doesn't retry; cache may say "processed" while DB doesn't.**

This is the worst class of webhook bug: the platform thinks delivery succeeded, the application thinks it failed, and there is no automated recovery path. Operators only discover the drift when a customer complains about being charged but not upgraded.

**Fix:** Distinguish "duplicate / known-processed" (200 OK, no retry) from "unprocessed / transient failure" (5xx, ask Stripe to retry):

```ts
try {
  await handleStripeWebhook(event);
  res.status(200).json({ received: true });
} catch (err) {
  logger.error('[Webhook] Processing error:', err);
  // 500 tells Stripe to retry with exponential backoff (up to 3 days).
  // The corrected processWebhookIdempotently (see CR-01) ensures the
  // idempotency record is removed if the handler failed, so retry
  // re-runs the handler instead of being blocked as a duplicate.
  res.status(500).json({ error: 'Processing failed, will retry' });
}
```

Pair this with CR-01's "delete on handler failure" branch so retries actually retry the work instead of being blocked as duplicates.

### CR-03: Internal error messages leak to API clients (information disclosure)

**File:** `apps/web/server/routes/subscriptions.ts:99-105, 131-146, 168-174`
**Issue:** All three endpoints surface raw `Error.message` strings to clients on 500:

```ts
} catch (err) {
  const message = err instanceof Error ? err.message : 'Failed to create checkout session';
  res.status(500).json({
    success: false,
    error: message,   // <- leaks internal exception text
  });
}
```

This violates the project security checklist (`~/.claude/rules/security.md` / `CLAUDE.md`: "Error messages don't leak sensitive data") and exposes:

- Stripe SDK errors with internal request IDs, partial customer IDs, account-level metadata
- Prisma errors that may include column names, constraint names, the literal failing row payload (e.g., on a unique-violation Prisma's default message includes the conflicting field+value)
- Misconfiguration messages like `'Stripe not configured'`, `'No Stripe customer found for user'` — useful for enumeration of which users have subscriptions

`subscriptions.ts:134` (`if (message.includes('No Stripe customer'))`) confirms the implementer already noticed that Stripe-customer-existence is a leak vector and tries to remap it to a friendlier 400 error — but only catches that one case; every other Stripe/DB error still leaks through.

**Fix:** Log the full error server-side, return a generic message to the client:

```ts
} catch (err) {
  logger.error('[subscriptions.checkout] failed', {
    userId: req.user?.userId,
    err: err instanceof Error ? { message: err.message, stack: err.stack } : err,
  });
  res.status(500).json({
    success: false,
    error: 'Failed to create checkout session',
  });
}
```

Apply identically to `/portal` and `/status`. Keep the existing remap-to-400 for the "No Stripe customer" sentinel, but switch from string matching to a typed sentinel error class (see WR-04).

### WR-01: `priceId || (billingCycle === 'annual' ? ... )` fallback is dead code after Zod validation

**File:** `apps/web/server/routes/subscriptions.ts:65-68`
**Issue:** The Zod schema requires `priceId` (line 23: `z.string().min(1, 'Price ID required')`). After `safeParse` succeeds, `result.data.priceId` is guaranteed to be a non-empty string. The `priceId || (billingCycle === 'annual' ? ... : ...)` fallback can never take the right-hand branch, so the `billingCycle` field is documented in the schema but never functional — clients passing only `billingCycle` get rejected at validation. This is misleading API surface (the schema implies billingCycle is a usable input) and the dead branch will silently rot if anyone makes `priceId` optional later.

**Fix:** Either make `priceId` optional in the schema and use `billingCycle` as the fallback path, or drop `billingCycle` from the schema entirely. The simpler fix:

```ts
const checkoutSchema = z.object({
  priceId: z.string().min(1, 'Price ID required'),
  // billingCycle removed — priceId is the source of truth
});

// ...
const { priceId } = result.data;
if (!isValidPriceId(priceId)) { /* ... */ }
```

If `billingCycle` is genuinely needed for UX, make `priceId` optional and require exactly one of the two via `.refine()`.

### WR-02: `prisma.config.ts` does not fail-fast when `DATABASE_URL` is unset

**File:** `apps/web/prisma.config.ts:32-34`
**Issue:**

```ts
datasource: {
  url: process.env["DATABASE_URL"],  // string | undefined
},
```

If `DATABASE_URL` is unset, this passes `url: undefined` into Prisma config. Prisma will either fail downstream with a confusing message ("datasource url required" or similar) or — depending on schema defaults — connect to an unintended database. The user-level rule `~/.claude/rules/golden-principles.md` §2 explicitly says "Access only via process.env, and throw immediately if not set." `apps/web/server/services/authService.ts:34-39` already follows this pattern for `JWT_SECRET`.

**Fix:**

```ts
const databaseUrl = process.env["DATABASE_URL"];
if (!databaseUrl) {
  throw new Error('DATABASE_URL environment variable is required (loaded by dotenv/config above)');
}

export default defineConfig({
  schema: "prisma/schema.prisma",
  migrations: { path: "prisma/migrations" },
  datasource: { url: databaseUrl },
});
```

### WR-03: `invoice.subscription as string` cast hides expanded-object case

**File:** `apps/web/server/services/stripeWebhookService.ts:184, 213`
**Issue:** `Stripe.Invoice.subscription` is typed `string | Stripe.Subscription | null`. The current code:

```ts
const subscriptionId = invoice.subscription as string;
if (!subscriptionId) return;
```

Handles `null` correctly (early return), but if the invoice was retrieved with `expand: ['subscription']` upstream — or Stripe ever changes the default representation — `subscriptionId` would be a `Stripe.Subscription` object. The truthiness check passes, then `findUserBySubscriptionId(subscriptionId)` issues a query like `WHERE stripeSubscriptionId = '[object Object]'`, returning no user, logging a false-positive "no user found" warning, and silently dropping the invoice update.

Same pattern applies to `subscription.customer as string` on lines 143, 158 of `handleSubscriptionUpdated`.

**Fix:**

```ts
const subscriptionId = typeof invoice.subscription === 'string'
  ? invoice.subscription
  : invoice.subscription?.id;
if (!subscriptionId) return;
```

Apply consistently to all `*.customer as string` and `*.subscription as string` casts in this file.

### WR-04: Error categorization via `message.includes('No Stripe customer')` is fragile

**File:** `apps/web/server/routes/subscriptions.ts:134`
**Issue:** The portal endpoint distinguishes "user has no Stripe customer yet" (400) from generic 500 by string-matching the error message:

```ts
if (message.includes('No Stripe customer')) {
  res.status(400).json({ ... });
}
```

The error is thrown in `subscriptionService.ts:99` as `throw new Error('No Stripe customer found for user')`. Any refactor of that string — e.g., adding a userId for diagnostics: `'No Stripe customer found for user ${userId}'` — would still match (substring), but renaming the message ("User has no billing account") would silently break the categorization, demoting an expected 400 into a 500 that leaks via CR-03.

**Fix:** Use a typed error class:

```ts
// In subscriptionService.ts
export class NoStripeCustomerError extends Error {
  constructor(userId: string) {
    super(`No Stripe customer found for user ${userId}`);
    this.name = 'NoStripeCustomerError';
  }
}

// Throw site
throw new NoStripeCustomerError(userId);

// Route handler
} catch (err) {
  if (err instanceof NoStripeCustomerError) {
    res.status(400).json({ success: false, error: 'No active subscription found' });
    return;
  }
  // ... generic 500 (per CR-03)
}
```

**File:** `apps/web/server/services/stripeWebhookService.ts:241-258, 263-280`
**Issue:**

### IN-01: `req.headers['stripe-signature'] as string` ignores multi-value-header case

**File:** `apps/web/server/routes/webhooks/stripe.ts:44`
**Issue:** Express types the header as `string | string[] | undefined`. The `as string` cast hides the array case. If a malicious / misbehaving proxy injects a duplicate `Stripe-Signature` header, `signature` becomes an array. The next-line truthiness check (`if (!signature)`) does not catch arrays (a non-empty array is truthy), and `stripe.webhooks.constructEvent(req.body, [array], secret)` will throw a typed signature-verification error that's caught by the existing 401 branch — so net safety is preserved, but the type cast is misleading and would mask a future bug if the catch branch were narrowed.

**Fix:**

```ts
const sigHeader = req.headers['stripe-signature'];
const signature = Array.isArray(sigHeader) ? sigHeader[0] : sigHeader;
if (!signature) { /* ... 400 ... */ }
```

### IN-05: `handleCheckoutCompleted` does not record the customer-creation race

**File:** `apps/web/server/services/stripeWebhookService.ts:104-132`
**Issue:** The function pulls `userId` from `session.metadata?.userId || session.client_reference_id`, then calls `getStripeSubscription(subscriptionId)`. If `getStripeSubscription` returns `null` (Stripe API hiccup), we log and return — but the user's checkout has already succeeded on Stripe's side. Stripe will not re-deliver `checkout.session.completed`, so the user is now charged but unlinked. The eventual `customer.subscription.created` event (also delivered) will hit `handleSubscriptionUpdated`, which tries `findUserBySubscriptionId` first (returns null because we never wrote it), then falls back to `findUserByCustomerId` — which also returns null because we never wrote `stripeCustomerId` either. Result: orphaned paying user.

**Fix:** When `getStripeSubscription` returns null, throw instead of silently returning. The webhook will then 500 (post-CR-02 fix) and Stripe will retry, giving us another chance to fetch the subscription. Alternatively, persist the customer ID immediately even if subscription fetch fails:

```ts
if (!subscription) {
  logger.error(`[Webhook] Could not fetch subscription ${subscriptionId}; persisting customer link only`);
  await prisma.user.update({
    where: { id: userId },
    data: { stripeCustomerId: customerId },
  });
  throw new Error(`Could not fetch subscription ${subscriptionId}`); // triggers retry
}
```

### IN-06: `default` case in `handleStripeWebhook` records unhandled events as processed

**File:** `apps/web/server/services/stripeWebhookService.ts:94-96`
**Issue:** `processWebhookIdempotently` records the event as processed even when the inner switch hits `default` (unknown event type). This is correct for Stripe's at-least-once-delivery model (we don't want unrelated event types replayed forever), but it means if a future code change adds a handler for, e.g., `customer.subscription.trial_will_end`, all historical events of that type that arrived during the unhandled period are permanently marked processed and will not back-fill. This is a general property of idempotency-keyed dispatchers, not a bug — but worth documenting.

**Fix:** Add a comment:

```ts
default:
  // Unknown event types are still recorded as processed to prevent
  // Stripe from retrying them forever. If you add a new handler, you
  // will not retroactively process events that arrived before the
  // handler existed — replay them via Stripe Dashboard if needed.
  logger.debug(`[Webhook] Unhandled event type: ${event.type}`);
```
