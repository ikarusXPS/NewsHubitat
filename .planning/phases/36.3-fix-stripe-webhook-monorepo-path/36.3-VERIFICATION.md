### Human Verification Required

#### 1. customer.subscription.resumed end-to-end against real subscription

**Test:** Pause then resume a real Stripe subscription with a registered live webhook URL. Specifically:
1. Register a live webhook URL pointing at the production/staging backend in the Stripe dashboard.
2. Create a subscription, force payment failure to reach `status=paused`, then resume payment to trigger `customer.subscription.resumed`.
3. Observe the webhook delivery in the Stripe dashboard event log.

**Expected:**
- Stripe dashboard shows webhook delivery `customer.subscription.resumed` with HTTP 200 response from the live backend.
- `ProcessedWebhookEvent` table in production DB contains a row with `eventType='customer.subscription.resumed'` for that event ID.
- The corresponding User row's `subscriptionStatus` transitions PAUSED -> ACTIVE and `pausedUntil` is cleared to null.
- Replaying the same event from the dashboard does not create a duplicate `ProcessedWebhookEvent` row.

**Why human:** The Stripe CLI does not expose a `customer.subscription.resumed` fixture (verified during Plan 04: `stripe trigger customer.subscription.resumed` is unsupported). Reaching that event via the Stripe API requires `subscription.status = paused` first, which itself requires a real failed-payment cycle — infeasible in a synthetic-fixture sandbox session. Plan 04 documented this CLI limitation explicitly in `36.3-04-SUMMARY.md` (Decision #1) and accepted structural coverage based on the handler's switch dispatching all 8 events through the same `processWebhookIdempotently` path proven 93 times. This is the one truly empirical gap; it is intentional and routed to production-readiness phase per the documented plan deviation.
