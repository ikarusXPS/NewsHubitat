## Fixed Issues

### CR-01: Webhook idempotency check is racy — handler can execute twice for the same event

**Files modified:** `apps/web/server/services/stripeWebhookService.ts`
**Commit:** `354561e`
**Applied fix:** Replaced the check-then-act sequence (Redis check -> DB check -> handler -> dual write) with an "insert-first, run-handler-second" pattern. The unique constraint on `ProcessedWebhookEvent.id` now acts as the cross-process lock against Stripe's at-least-once delivery. P2002 (duplicate-PK) means another delivery already claimed this event — re-cache and return duplicate. On handler exception, the row is deleted so Stripe's retry can re-process (without delete, the event would be poisoned: marked processed, side-effect never applied). Preserves the Phase 36.2 Redis-primary + DB-fallback dual-storage semantics.

**Note:** This is a logic fix — verified via re-read and full test suite (1289 passing). Concurrent-delivery semantics are not exercised by existing tests; recommend a phase verifier review before declaring fully-verified for production. Marking as `fixed: requires human verification` for safety.

### CR-02: Webhook route silently swallows ALL processing errors with HTTP 200

**Files modified:** `apps/web/server/routes/webhooks/stripe.ts`
**Commit:** `daa0dd8`
**Applied fix:** Changed the outer `catch` block from `res.status(200).json({ received: true, warning: 'Processing error logged' })` to `res.status(500).json({ error: 'Processing failed, will retry' })`. Combined with CR-01's lock-release-on-failure, retries now actually re-run the handler instead of being blocked as duplicates. The justifying comment was rewritten to explain the new contract: 200 only for success or known-duplicate; 500 to signal "not yet recorded, please retry".

### CR-03: Internal error messages leak to API clients

**Files modified:** `apps/web/server/routes/subscriptions.ts`
**Commit:** `cd5f24c`
**Applied fix:** All three subscription endpoints (`/checkout`, `/portal`, `/status`) now log the full error server-side via winston (with `userId`, `message`, `stack`) and return a fixed generic message to the client. Imported `logger from '../utils/logger'` for this. The "No Stripe customer" sentinel match was retained on `/portal` for this commit so the 400 behavior is unchanged during the CR-03 -> WR-04 transition; WR-04 then replaces the string match with a typed sentinel. No client-facing API shape change beyond the error string.

### WR-01: `priceId || (billingCycle === 'annual' ? ... )` fallback is dead code

**Files modified:** `apps/web/server/routes/subscriptions.ts`
**Commit:** `33379ab`
**Applied fix:** Dropped `billingCycle` from the Zod schema and removed the unreachable fallback expression. `priceId` is now the sole source of truth (already required as `z.string().min(1)`). Existing e2e callers that send both fields continue to validate (Zod's default `z.object()` strips unknown keys). Verified via `apps/web/e2e/subscription.spec.ts:188` which sends `{ priceId, billingCycle }` and accepts `[400, 500, 503]` outcomes — unaffected by this change.

### WR-02: `prisma.config.ts` does not fail-fast when `DATABASE_URL` is unset

**Files modified:** `apps/web/prisma.config.ts`
**Commit:** `8723fa4`
**Applied fix:** Added an explicit guard right after the `dotenv/config` import: read `process.env["DATABASE_URL"]` into a `const`, throw if empty, then pass to `defineConfig({ datasource: { url: databaseUrl } })`. Mirrors the JWT_SECRET fail-fast guard in `authService.ts` and complies with golden-principles §2 ("Access only via process.env, and throw immediately if not set").

### WR-03: `invoice.subscription as string` cast hides expanded-object case

**Files modified:** `apps/web/server/services/stripeWebhookService.ts`
**Commit:** `032130a`
**Applied fix:** Added an `extractStripeId(value)` helper that handles Stripe's `string | T | null` union. Replaced all four `as string` cast sites (also fixed two more bonus sites in `handleCheckoutCompleted` not explicitly called out in the review but covered by the WR-03 directive "Apply consistently to all `*.customer as string` and `*.subscription as string` casts in this file"). For `invoice.subscription`, the Stripe SDK v22 type definitions dropped this field (deprecated in newer API versions, but still sent on the project's pinned `2024-12-18.acacia` wire payload) — used a typed `as unknown as { subscription?: string | Stripe.Subscription | null }` cast to keep the runtime access explicit and survive strict typecheck.

### WR-04: Error categorization via `message.includes('No Stripe customer')` is fragile

**Files modified:** `apps/web/server/services/subscriptionService.ts`, `apps/web/server/routes/subscriptions.ts`
**Commit:** `8960f35`
**Applied fix:** Defined `class NoStripeCustomerError extends Error` in `subscriptionService.ts`, exported alongside `SubscriptionService`. Throw site (`createPortalSession`) now uses `throw new NoStripeCustomerError(userId)` (carrying userId for logs but locked by type, not string). Catch site (`/portal` route) now uses `if (err instanceof NoStripeCustomerError)` instead of `message.includes('No Stripe customer')`. No public API change: client still gets the same `{400, 'No active subscription found'}` payload.
