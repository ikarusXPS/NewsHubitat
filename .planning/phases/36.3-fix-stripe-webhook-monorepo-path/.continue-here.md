# Anti-Pattern Registry — Phase 36.3

> Future plan-phase / execute-phase orchestrators must read this file before scoping a phase.
> Entries here apply when phase 36.3 (or downstream phases reading this directory) is active.

| Anti-pattern | Severity | Manifestation | Structural prevention |
|--------------|----------|---------------|------------------------|
| Writing files to root `server/`, `prisma/`, or `src/` instead of `apps/web/...` | blocking | Phase 36-02 wrote 3 webhook source files to root `server/` instead of `apps/web/server/`, leaving the live backend with a 404 on every Stripe event for ~2 days. Three milestone-36 incidents total (36.1 schema additions claimed but never written; 36.2 schema models claimed but never written; 36.3 webhook files written to wrong path). | Post-36.3, root `server/`, `prisma/`, `src/`, `tsup.config.ts`, `prisma.config.ts` are deleted — they cannot be a write target by accident. Future executor agents reading this file MUST verify their `<files_modified>` paths begin with `apps/web/`, `packages/`, or top-level config files (`package.json`, `pnpm-workspace.yaml`, `.github/`, etc.) before commit. |

## How to comply

When planning or executing any phase that touches backend / schema / frontend code:

1. Confirm `<files_modified>` paths begin with one of: `apps/web/`, `packages/`, `.github/`, `.planning/`, `pnpm-workspace.yaml`, `package.json`, `tsconfig.base.json` (if it survived 36.3 sweep), `docker-compose.yml`.
2. NEVER write to root `server/`, `prisma/`, `src/`, `prisma.config.ts`, or `tsup.config.ts`. These were deleted in 36.3-03; recreating them is a regression.
3. Run prisma commands ONLY from `apps/web/` (per CLAUDE.md lines 47-49). The deletion of root `prisma.config.ts` enforces this — running `npx prisma db push` from repo root will fail with "no schema found" rather than silently load a stale duplicate (the failure mode that nearly dropped the ApiKey table in Phase 36.2-03).
