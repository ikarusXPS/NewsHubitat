# Phase 36.3: Fix Stripe Webhook Monorepo Path - Context

**Gathered:** 2026-04-28
**Status:** Ready for planning

<domain>
## Phase Boundary

Restore Stripe webhook handler + subscription routes to the live monorepo backend, then sweep all stale root-level orphan trees that have accumulated from Phase 36-02 and earlier. Phase 36-02 (commits `0ec4634`, `a32f285`, `6049d1e` on 2026-04-26) wrote the webhook service, route, and subscription routes — plus `index.ts` mount edits — into root-level `server/` instead of `apps/web/server/`. Live backend (run by `pnpm dev:backend` from `apps/web/server/index.ts`) returns 404 for every Stripe event despite signing secret + raw body parser being in place.

This phase also closes the broader "orphan root tree" technical debt that surfaced during 36.2-03 (root `prisma/schema.prisma` + `prisma.config.ts` nearly dropped the `ApiKey` table when `prisma db push` resolved against the wrong schema). After 36.3, the monorepo has exactly one canonical location for backend code (`apps/web/server/`), schema (`apps/web/prisma/`), and frontend (`apps/web/src/`).

**In scope:**
- Relocate 3 source files from `server/` → `apps/web/server/` via `git mv` (preserves history): `services/stripeWebhookService.ts`, `routes/webhooks/stripe.ts`, `routes/subscriptions.ts`
- Diff-only edits to `apps/web/server/index.ts`: import + mount `stripeWebhookRouter` BEFORE `express.json()` (line 109); import + mount `subscriptionRoutes` AFTER `/api/teams` (line 170) with `authMiddleware`; no global rate limiter on subscription routes
- Wholesale delete of root orphan trees: `server/`, `prisma/`, `src/`, `tsup.config.ts`, `prisma.config.ts`
- End-to-end verification via `stripe trigger` against all 8 events the handler subscribes to + DB-row check for `ProcessedWebhookEvent` idempotency
- Anti-pattern entry in `.planning/phases/36.3-fix-stripe-webhook-monorepo-path/.continue-here.md` flagging "no writes to root server/prisma/src/ — apps/web/ is canonical"

**Out of scope (belongs in future phases):**
- Refactoring webhook event handlers themselves (the orphan service code is functional; only the mount path is broken)
- Adding new event types beyond the 8 already subscribed
- Production webhook deployment / Stripe dashboard config (env vars, webhook URL registration in Stripe — handled by the production-readiness phase)
- Subscription route rate-limiting (deferred until abuse signal observed; webhook + Stripe API already have their own limits)

</domain>

## Implementation Decisions

### Source File Relocation

- **D-01:** Use `git mv` (not copy-then-delete) to relocate the 3 source files. Each file move is its own atomic commit so `git log --follow` traces history through the rename. Files (3 total): `server/services/stripeWebhookService.ts` → `apps/web/server/services/stripeWebhookService.ts`; `server/routes/webhooks/stripe.ts` → `apps/web/server/routes/webhooks/stripe.ts`; `server/routes/subscriptions.ts` → `apps/web/server/routes/subscriptions.ts`.
- **D-02:** All inner imports survive the move unchanged. Verified pre-move: `stripeWebhookService.ts` imports `../db/prisma`, `./cacheService`, `./subscriptionService`, `../utils/logger`, `../config/stripe` — all resolve correctly post-relocation because the directory layout under `apps/web/server/` mirrors root `server/` exactly. `routes/subscriptions.ts` imports `../services/authService`, `../services/subscriptionService`, `../config/stripe` — same. `routes/webhooks/stripe.ts` imports `../../services/stripeWebhookService`, `../../utils/logger` — same. No source-file edits needed during relocation.

### `apps/web/server/index.ts` Mount Strategy

- **D-03:** Diff-only edits to the live `apps/web/server/index.ts` — do NOT replace it with the orphan root copy. Reason: root `server/index.ts` is a stale Phase 35-pre snapshot; it lacks `apiKeyRoutes`, `publicApiRoutes`, `apiKeyAuth`, `apiKeyLimiter`, and `X-API-Key` CORS header — replacing the live file would silently regress all of Phase 35.
- **D-04:** Webhook mount placement — insert at line 108 of the current live `apps/web/server/index.ts` (immediately BEFORE the `app.use(express.json(...))` block at line 110). Two-line insertion: import at the top with the other route imports; mount line `app.use('/api/webhooks/stripe', stripeWebhookRouter);` before `express.json()`. The `express.raw({ type: 'application/json' })` is already declared INSIDE the webhook route file (`routes/webhooks/stripe.ts:35`) so the parent app does not need a second raw-body middleware.
- **D-05:** Subscription route mount placement — insert AFTER the `/api/teams` mount (live `apps/web/server/index.ts:170`) and BEFORE the `/api/keys` mount (line 173). Mount: `app.use('/api/subscriptions', authMiddleware, subscriptionRoutes);`. `authMiddleware` is exported from `apps/web/server/services/authService.ts:559`. NO global rate limiter — per-user authMiddleware gating + Stripe API native rate-limiting are sufficient. Webhook abuse vector is signature-verification-gated; subscription-creation abuse vector is auth-gated.
- **D-06:** Webhook route bypasses `compression`, `etagMiddleware`, `serverTimingMiddleware`, `metricsMiddleware`, `queryCounterMiddleware` because it's mounted before any of them at line 108 (the live order: cors → compression → serverTiming → etag → metrics → queryCounter → passport → **WEBHOOK** → express.json → debug → routes). Wait — actually re-reading, the live `apps/web/server/index.ts` has compression/etag/metrics/queryCounter at lines 89–101 BEFORE `express.json()` at line 110. So the webhook (inserted at line 108) WILL go through compression/etag/metrics. Acceptable for a 200 OK with a small JSON response, but compression on 200-byte responses is harmless (compression's `threshold: 1024` skips small bodies anyway). If subtle issues arise, the executor can move the webhook mount earlier (right after CORS at line 86) — but that's a deviation, not a default.

### End-to-End Webhook Verification

- **D-09:** Run `stripe listen --forward-to http://localhost:3001/api/webhooks/stripe` in one terminal; trigger all 8 events the handler subscribes to from another terminal: `stripe trigger checkout.session.completed`, `customer.subscription.created`, `customer.subscription.updated`, `customer.subscription.deleted`, `customer.subscription.paused`, `customer.subscription.resumed`, `invoice.paid`, `invoice.payment_failed`. Re-triggering the same event should NOT create a duplicate row (the handler returns the cached idempotency response).

This means the live `apps/web/server/index.ts` `express.json()` block at line 110 needs no change — only the webhook mount needs to land before it.
- **`authMiddleware` export** — Already at `apps/web/server/services/authService.ts:559`. Subscription routes import it from `../services/authService` (relative path survives the move).
- **`STRIPE_CONFIG`** — Already at `apps/web/server/config/stripe.ts` (Phase 36-01, refactored in 36.2-03 to re-export Prisma enums). Subscription routes' price-ID whitelist (`STRIPE_CONFIG.priceIds.monthly`, `.annual`) is intact.

The webhook handler's `prisma.processedWebhookEvent.findUnique({ where: { id: eventId }})` and `.create({ data: { id, eventType }})` calls work end-to-end the moment the route is mounted.

### Established Patterns
- **Mount order discipline** — Live `apps/web/server/index.ts` mounts routes in a documented order: CORS (line 70) → compression (89) → serverTiming (92) → etag (95) → metrics (98) → queryCounter (101) → passport (104) → **insert webhook here (108)** → express.json (110) → routes. - **Idempotency via dual-storage** — The handler's pattern (Redis primary, DB backup) is consistent with the existing webhook idempotency convention (Phase 22 SMTP webhooks use the same model). Reuse, don't reinvent.
