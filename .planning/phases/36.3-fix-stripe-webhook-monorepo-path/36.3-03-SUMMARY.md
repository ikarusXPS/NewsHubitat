## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking] `prisma db push` failed with "datasource.url required in your Prisma config file when using prisma db push" after deleting root prisma.config.ts**

- **Found during:** Task 2 step 4f (D-08 prisma db push idempotency check, after the sweep commit 651ce93 was already on HEAD)
- **Issue:** The orphan sweep deleted root `prisma.config.ts`, which was load-bearing in two ways:
  1. **Bug:** Its `schema: "prisma/schema.prisma"` resolved relative to its own (root) directory, hitting the orphan duplicate root prisma/ tree — this was the Phase 36.2-03 ApiKey-drop incident cause and the reason it was on the deletion list.
  2. **Load-bearing:** Its `datasource: { url: process.env["DATABASE_URL"] }` was the only datasource URL source for Prisma CLI commands. apps/web/prisma/schema.prisma's datasource block intentionally has no `url = env("DATABASE_URL")` line — the existing comment in the schema says "URL defined in prisma.config.ts (Prisma 7+)". Phase 36.2 deliberately picked the prisma.config.ts pattern rather than declaring URL in schema.
- **Fix:** Created `apps/web/prisma.config.ts` (workspace-local, NOT at the repo root) with the same datasource declaration. The 36.2-03 / 36.3-03 anti-pattern is specifically a ROOT-LEVEL prisma.config.ts (which resolves schema relative to its own directory and silently picks up the orphan duplicate); a workspace-local config in apps/web/ resolves `schema: "prisma/schema.prisma"` to the canonical apps/web/prisma/schema.prisma — correct in every invocation regardless of cwd.
- **Files modified:** `apps/web/prisma.config.ts` (created, 35 lines)
- **Commit:** `a69bbff fix(36.3-03): create apps/web/prisma.config.ts for Prisma 7 datasource resolution`
- **Re-verify:** typecheck still 0; 1289/1289 tests still PASS; prisma validate "schema valid"; prisma db push "already in sync".

**2. [Filesystem cleanup, non-deviation] Empty directories + gitignored artifacts left after `git rm -rf`**

- **Found during:** Task 2 final acceptance criteria check (`test ! -d server` and `test ! -d src` initially failed)
- **Issue:** `git rm -rf` removes git-tracked files. After it ran:
  - `server/routes/webhooks/` was an empty directory (Plan 01's earlier `git mv` had vacated it; Windows didn't auto-prune empty dirs).
  - `src/generated/prisma/` contained gitignored Prisma-generated TypeScript files (per `.gitignore` line 31: `/src/generated/prisma`) that `git rm` never knew existed.
- **Fix:** Filesystem-level `rm -rf server src` (no commit needed — no tracked files affected; the previous commit already captured all tracked deletions).
- **Verification:** Grep across `apps/` confirmed every reference to `src/generated/prisma` resolves to `apps/web/src/generated/prisma/` (the canonical 36.2-03 client), NOT root `src/generated/`. Live code is unaffected.
- **Why this is documented as a non-deviation:** the plan's deletion list specified the root-level paths; cleaning up untracked-but-still-on-disk artifacts to satisfy `test ! -d` acceptance is a post-commit cleanup, not a deviation from the plan's intent.
