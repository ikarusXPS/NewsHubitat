## Architecture Note

`TeamSettingsModal` uses an **inner/outer component split**: the outer `TeamSettingsModal` gates on `isOpen` and returns null when closed; the inner `TeamSettingsForm` only mounts when the modal is open. This pattern:
- Eliminates the need for a state-reset `useEffect` (form state re-initialises from props on every open)
- Avoids the React Compiler `setState-in-effect` lint error that affected `DeleteTeamModal.tsx` (pre-existing issue, not introduced here)
- `DeleteTeamModal.tsx` has the same pattern issue at line 36 — this plan demonstrates the fix but does not touch that file (out of scope)

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Enhancement] Inner/outer component split instead of setState-in-effect**
- **Found during:** Task 1 implementation
- **Issue:** The plan's provided code used `useEffect(() => { if (isOpen) { setName(); setDescription(); } }, [isOpen, ...])` — the React Compiler's `react-compiler/react-hooks` rule flags this as an error ("Calling setState synchronously within an effect can trigger cascading renders"). The same issue exists in `DeleteTeamModal.tsx` (line 36) as a pre-existing error.
- **Fix:** Split into `TeamSettingsForm` (inner, mounts only when `isOpen=true`) + `TeamSettingsModal` (outer wrapper, returns null when closed). This avoids the effect entirely — state initializes from props naturally on each mount.
- **Files modified:** `apps/web/src/components/teams/TeamSettingsModal.tsx`
- **Tests impact:** All 8 tests pass unchanged.

None of the pre-existing lint errors in other files (VirtualizedList, VirtualizedGrid, coverage/block-navigation.js) are in scope for this plan.
