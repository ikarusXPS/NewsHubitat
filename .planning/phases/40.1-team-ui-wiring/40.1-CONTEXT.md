**Gathered:** 2026-04-30

<domain>
## Phase Boundary

This is **debt-payback** for Phase 28 (Team Collaboration, shipped in v1.4). The 4 debug sessions identified UI components that existed but were never wired to existing backend infrastructure. A codebase audit on 2026-04-30 (the day this CONTEXT was written, 5 days after the debug sessions) revealed that **3 of 4 issues have already been silently wired** by subsequent commits — but no tests prove the wiring works, no debug session has been marked resolved, and one sub-feature remains genuinely missing.

The phase delivers:
1. **Verification** that the silent fixes work end-to-end
2. **Test coverage** so the wiring cannot silently regress (this is what the original Phase 28 lacked — that's why the debt happened)
3. **The one truly missing piece**: `TeamSettingsModal` for editing team name/description (backend `PATCH /api/teams/:teamId` + `useUpdateTeam` hook already exist, no UI consumes them)
4. **Resolution housekeeping**: archive the 4 debug session files to `.planning/debug/resolved/`

**Out of scope:**
- New team features (member role changes via UI, ownership transfer, team avatars, etc.) — backend has the endpoint for some, but those are v1.7+ work
- Any backend changes — every endpoint and hook the UI needs already exists
- TeamSettingsModal as the *delete entry point* — the existing `Trash2` icon button on `TeamDashboard.tsx:121-128` (visible to owners only) already handles delete. The modal added here is for *editing* (name/description) only.

</domain>

<decisions>
## Implementation Decisions

### D-01: Verification-first scoping
- **Decision:** Treat the 3 silently-fixed flows (#2, #3, #4) as "must verify, must test, do not re-implement". The wiring is already in place; we add tests around it and confirm via UAT, not new code.
- **Why:** Re-implementing working code creates regressions. Tests are the durable artifact that prevents this debt from recurring.

### D-02: TeamSettingsModal — minimal first cut
- **Decision:** New component `apps/web/src/components/teams/TeamSettingsModal.tsx`. Editable fields: team `name` (required, ≤50 chars per existing schema) and `description` (optional, ≤500 chars). Wires to existing `useUpdateTeam(teamId)` hook (`apps/web/src/hooks/useTeams.ts:189`). Visible to owners and admins (matches backend `requireTeamRole('owner', 'admin')` on `PATCH /:teamId`).
- **Trigger:** New gear icon (`Settings` from `lucide-react`) in `TeamDashboard.tsx` header action group, placed BEFORE the existing `Trash2` button. Visible to owners + admins. The Trash2 button stays put — it remains the delete entry point.
- **NOT in scope for this phase:** Role management (promoting members, transferring ownership) — Phase 28's UI-SPEC mentioned it but it's a separate feature with its own backend gaps; defer to v1.7.
- **Why:** Smallest delta that closes the actual debt. The original `TeamSettingsModal` UI-SPEC bundled name/description AND role management — splitting those keeps this phase finite.

### D-03: Unit tests for wired components
- **Decision:** Add Vitest unit tests for the 4 wired components, focused on the wiring logic (props flowing through, hooks called, conditional rendering by role). Use existing patterns from `apps/web` test files (Vitest + React Testing Library).
- **Files:**
  - `apps/web/src/components/BookmarkButton.test.tsx` — team dropdown renders when `useTeams()` returns teams; `useAddTeamBookmark` called on team selection; falls back to personal toggle when no teams
  - `apps/web/src/components/teams/TeamBookmarkCard.test.tsx` — uses `articleUrl` for href when provided; falls back to `/article/${articleId}` when undefined; `target="_blank"` always present; remove button visible per role
  - `apps/web/src/components/teams/PendingInviteList.test.tsx` — renders invite rows from hook; cancel button visible for admin+; cancel mutation called on click
  - `apps/web/src/components/teams/DeleteTeamModal.test.tsx` — confirmation gate works; delete mutation called; navigates away on success
  - `apps/web/src/components/teams/TeamSettingsModal.test.tsx` — form validation, mutation called with name/description, closes on success
- **Coverage target:** Each new test file gets the affected component to ≥80% line coverage. Project gate is 75% branch / 80% line (per CLAUDE.md).
- **Why:** The whole reason this debt existed is no test proved the wiring. Tests are the receipts.

### D-04: E2E coverage
- **Decision:** Extend `apps/web/e2e/teams.spec.ts` with 4 new authenticated tests covering exactly the 4 debug-session flows. Mark them serial via `test.describe.configure({ mode: 'serial' })` since they share team state.
- **Tests to add:**
  1. **Save bookmark to team** — visit a news card, click bookmark, choose a team from dropdown, navigate to team dashboard, see the bookmark with "Added by [user]" attribution.
  2. **Open team bookmark in new tab** — same setup, on team bookmarks tab, the article anchor has `target="_blank"` and a non-empty `href` ending in the article URL (not `/article/{id}`).
  3. **View pending invites tab** — owner navigates to team dashboard as admin, switches to "Pending Invites" tab, sends an invite via `InviteModal`, sees the new invite appear in the list with cancel button.
  4. **Delete team** — owner clicks Trash2 button, modal opens, confirms, redirects away, team gone from `useTeams()` switcher.
- **Why "wait for `domcontentloaded`, not `networkidle`"** (CLAUDE.md E2E conventions) — Socket.io polling never lets network go idle.
- **Skip-edit settings test:** Only add an E2E for `TeamSettingsModal` if D-02 ships in the same wave; otherwise unit test alone is sufficient.

### D-05: TeamSettingsModal trigger placement
- **Decision:** Settings gear icon goes in `TeamDashboard.tsx:107-129` action group, between the "Invite Member" button and the existing "Delete Team" Trash2 button. Visible when `team.role === 'owner' || team.role === 'admin'` (`canInvite` already encodes this).
- **Why:** Reuses the existing role-gated action area; no header redesign needed.

### D-06: No backend changes
- **Decision:** Zero changes under `apps/web/server/`. Every endpoint and validation schema needed already exists (`PATCH /api/teams/:teamId` for edit, `DELETE /api/teams/:teamId`, `GET /api/teams/:teamId/bookmarks` with article enrichment, `GET /api/teams/:teamId/invites`, `POST /api/teams/:teamId/bookmarks`).
- **Why:** Phase 40.1 is debt-payback, not new feature work. Touching backend invites scope creep (and likely won't pass UAT in one phase).

### D-07: Debug session archival
- **Decision:** After verification (UAT) passes, move all 4 files from `.planning/debug/` to `.planning/debug/resolved/{date}-{slug}.md` (date = the day they were diagnosed, 2026-04-25, NOT the resolution date — the project convention per existing resolved files). Update each file's `status: diagnosed` → `status: resolved` and fill `verification:` and `files_changed:` blocks.
- **Why:** STATE.md insertion narrative explicitly mentions "archive 4 debug sessions to `.planning/debug/resolved/`" as the goal. Keeps `.planning/debug/` as a pure inbox for unresolved.

### Claude's Discretion
- Test file structure within Vitest (describe/it nesting, fixture placement) — match nearest existing test file
- TeamSettingsModal styling (colors, spacing, animation) — match `DeleteTeamModal.tsx` and `InviteModal.tsx` patterns (glass-panel, framer-motion, the cyan accent system from CLAUDE.md)
- i18n keys for new modal — use the `teams` namespace, follow existing key patterns (`teams.settings.title`, `teams.settings.nameLabel`, etc.). German + French translations come from i18next default-fallback at minimum; full translation in next i18n sweep
- Whether to refetch on settings save vs optimistic update — `useUpdateTeam` already invalidates `['teams']` and `['team', teamId]` queries; rely on that

## Specific Ideas

- TeamSettingsModal mirrors `DeleteTeamModal.tsx` structure (header with Settings icon, form body, action footer) but its "destructive" red palette swaps for the cyan accent palette since it's a non-destructive action.
- The settings gear icon trigger is `<Settings className="h-5 w-5" />` from `lucide-react` (already used elsewhere in the app — verify with grep).
- For the "open in new tab" E2E, assert `await locator.getAttribute('target') === '_blank'` AND `await locator.getAttribute('href')` does NOT match `/article/${articleId}` (so we know the join enriched the data, not the fallback path).
- For BookmarkButton unit test, the easiest mock is `vi.mock('../hooks/useTeams', () => ({ useTeams: () => ({ data: [{ id, name, role }] }) }))` — pattern matches existing project mocks.
- Debug session archival: rename to `.planning/debug/resolved/2026-04-25-{slug}.md`. The 2026-04-25 prefix matches the diagnosis date (project convention is the date the debug session was created, not when it was resolved — confirm by listing existing resolved files before committing).
