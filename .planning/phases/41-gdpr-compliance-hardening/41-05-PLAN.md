# Plan 41-05: Einwilligungsnachweis (TOM-FIX-03)

## Goal

Erfülle DSGVO Art. 7 Abs. 1 (Nachweispflicht) durch Server-Side Audit-Log der Consent-Bestätigungen. Aktuell steht der Consent-State nur in `localStorage` (`newshub-consent`) — unzureichend für Aufsichtsbehörde-Audit oder rechtlichen Streitfall. Neu: jede Bestätigung wird mit `bannerVersion` + `userAgent` + `ipHash` + `consents` + Timestamp serverseitig persistiert.

## Schema-Migration

```prisma
model ConsentEvent {
  id            String   @id @default(cuid())
  userId        String?  // null bei Pre-Login
  sessionId     String   // Browser-Session-ID (anonymer Cookie / fallback UUID)
  bannerVersion String   // z.B. "1.0.0"
  consents      Json     // { essential: true, preferences: bool, analytics: bool, marketing: bool }
  userAgent     String   @db.VarChar(500)
  ipHash        String   // SHA-256 — kein Klartext
  createdAt     DateTime @default(now())

  @@unique([sessionId, bannerVersion])  // Idempotenz
  @@index([userId])
  @@index([createdAt])
}
```

Retention: **3 Jahre** (BGB §195 Verjährungsfrist) — Cleanup-Job ergänzen.

## Steps

1. `schema.prisma` ergänzen + `prisma db push` (Dev)
2. `apps/web/src/lib/consent.ts` mit `export const CONSENT_BANNER_VERSION = '1.0.0'`
3. `apps/web/server/services/consentService.ts` mit `recordConsent({ userId, sessionId, bannerVersion, consents, userAgent, ip })` — IP wird vor Speicherung SHA-256-gehasht
4. `apps/web/server/routes/consent.ts` mit `POST /api/consent` (Zod-validiert, idempotent via `@@unique([sessionId, bannerVersion])`)
5. `apps/web/server/index.ts` Route mounten (vor `/api/auth` Stack), Rate-Limit 10 req/min pro IP
6. `apps/web/src/contexts/ConsentContext.tsx` ruft beim Akzeptieren `fetch('/api/consent', {...})` mit aktueller `CONSENT_BANNER_VERSION` auf
7. `ConsentBanner.tsx`: wenn `localStorage.newshub-consent.bannerVersion` < aktuelle Version → Banner erneut anzeigen mit Hinweis "Wir haben unsere Cookie-Richtlinie aktualisiert"
8. `cleanupService.ts` ergänzen: täglicher Job löscht `ConsentEvent` älter als 3 Jahre
9. `docs/legal/CONSENT-VERSIONS.md` mit Versions-History anlegen:
   ```markdown
   | Version | Datum | Änderung |
   |---------|-------|----------|
   | 1.0.0   | 2026-05-05 | Erste versionierte Banner-Auslieferung |
   ```
10. `openapi/schemas.ts` ConsentSchema ergänzen → `pnpm openapi:generate`

## Verification

- Unit-Test `consentService.test.ts`: IP wird gehasht, nicht im Klartext gespeichert
- Integration-Test: `POST /api/consent` zweimal mit gleichem `(sessionId, bannerVersion)` → einmal 201, einmal 200 (idempotent), keine doppelten Rows
- Manueller Browser-Test: Consent geben → DevTools Network → `POST /api/consent` mit Status 201 → DB-Row sichtbar in Prisma Studio
- Banner-Version-Bump simulieren: `CONSENT_BANNER_VERSION = '1.1.0'` → Banner erscheint erneut

## Out of scope

- DSGVO Art. 7 Abs. 3 (Widerruf-Mechanismus) — bereits über `ConsentBanner`-Reset abgedeckt; ggf. eigener Endpoint später
- Anonymized-Analytics über Consent-Akzeptanzraten (möglich, aber kein Audit-Erfordernis)
