# Plan 41-02: AVV-Tracking-Infrastruktur (ORG-01)

## Goal

Lege ein Tracking-Gerüst für die 17 ausstehenden AVV-Unterzeichnungen an. Jeder Anbieter bekommt eine eigene Markdown-Datei `docs/legal/avv/{vendor}.md` mit Status-Tracking. Sobald ein AVV unterzeichnet ist, wird in beiden Stellen aktualisiert: (a) Vendor-Tracker-File, (b) `PROCESSING-RECORDS.md` Sektion 4 Status-Spalte (☐ Ausstehend → ☑ Unterzeichnet YYYY-MM-DD).

## Status: BLOCKED

Externe AVV-Beschaffung läuft parallel — diese Plan-Datei ist ausführbar (legt nur Tracker-Files an), aber die Status-Spalten-Updates passieren über die Zeit, sobald AVVs unterzeichnet zurückkommen.

## Template (`docs/legal/avv/_template.md`)

```markdown
# AVV — {Anbieter-Name}

## Status

- **Unterzeichnet:** ☐ Nein / ☑ Ja, am YYYY-MM-DD
- **AVV-Version:** {z.B. v2.3 / 2025-01-15}
- **Beziehbar via:** {URL zum Vendor-Portal oder PDF-Storage}
- **Aufbewahrung des unterzeichneten AVVs:** {Pfad/Cloud-Storage}

## Vertragsgegenstand

{1-2 Sätze: welche Datenkategorien aus PROCESSING-RECORDS.md werden verarbeitet}

## Kritische Klauseln

- **Unterauftragsverarbeiter:** {Vendor's Liste, ggf. Link}
- **Drittlandübermittlung:** {USA / EU / sonstiges} — {SCCs Modul X / EU-US DPF / keine}
- **Audit-Rechte:** {beschrieben in §X}
- **Löschung bei Vertragsende:** {beschrieben in §X}

## Notizen

{Auffälligkeiten, Verhandlungspunkte, Verzögerungen}

## Änderungs-Log

| Datum | Aktion | Bearbeiter |
|-------|--------|-----------|
```

## Steps

1. `docs/legal/avv/README.md` mit Übersichts-Tabelle aller 17 Tracker erstellen
2. `docs/legal/avv/_template.md` mit obigem Template erstellen
3. 17 Tracker-Files (siehe `files_modified`) je vom Template ableiten — initial alle mit Status "☐ Nein"
4. `PROCESSING-RECORDS.md` Sektion 4 erweitern um Spalte "AVV-Datei" mit Link zum jeweiligen Tracker
5. Fortlaufend (nicht in einem Commit): Bei jeder Unterzeichnung Status-Update in Tracker + Sektion-4-Tabelle
