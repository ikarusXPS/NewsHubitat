# Plan 41-06: Transfer-Impact-Assessment pro US-Anbieter (ORG-03)

## Goal

Schrems-II-Compliance: Pro US-Drittanbieter eine TIA-Datei mit Risiko-Bewertung und Garantie-Dokumentation. Initial-Versionen werden recherchebasiert verfasst (Draft); rechtsverbindliche Final-Stempel braucht Anwalts-Review (separater Schritt).

## Template (`docs/legal/tia/_template.md`)

```markdown
# Transfer Impact Assessment — {Anbieter}

**Status:** ☐ Draft / ☐ Reviewed by counsel / ☐ Approved
**Erstellt:** YYYY-MM-DD
**Letzte Überprüfung:** YYYY-MM-DD

## 1. Übermittlungs-Beschreibung

- **Daten-Kategorien:** {aus PROCESSING-RECORDS.md Sektion 2 Verweis}
- **Volumen:** {z.B. ~10k Events/Tag}
- **Frequenz:** {permanent / event-getrieben / on-demand}
- **Empfänger:** {Anbieter + Rechtsperson, z.B. "Stripe Inc., 510 Townsend St, San Francisco"}

## 2. Rechtsgrundlage der Übermittlung

- **DSGVO Art. 46:** ☐ SCCs (Module 2 — Controller-to-Processor) / ☐ EU-US Data Privacy Framework
- **Vendor zertifiziert unter EU-US DPF?** {Ja/Nein — Link zu dataprivacyframework.gov Liste}

## 3. Risiko-Bewertung (Schrems II)

- **US-Surveillance-Reach:**
  - FISA 702 (relevant?): {Ja/Nein/Begründung}
  - EO 12333 (relevant?): {Ja/Nein/Begründung}
- **Sektoren-Spezifika:** {z.B. "Stripe ist Finanz-Provider, FISA 702 für Bulk-Collection unwahrscheinlich"}

## 4. Zusätzliche Garantien

- ☐ Ende-zu-Ende-Verschlüsselung
- ☐ Pseudonymisierung im Transit (z.B. `ipHash` statt Klartext)
- ☐ Zugriffsbeschränkung beim Empfänger (Vendor publiziert Access-Audit)
- ☐ Kurzlebige Tokens / TTL
- ☐ Log-Scrubbing (siehe Plan 41-07)

## 5. Schlussfolgerung

{Übermittlung zulässig? Knappe Begründung. Alternative wenn nicht: EU-Anbieter, Self-Host, Verzicht.}

## 6. Review-Schedule

Jährliche Überprüfung; bei Schrems III oder DPF-Aussetzung sofort.
```

## Steps

1. `_template.md` mit obigem Skelett erstellen
2. 17 TIA-Files je vom Template ableiten (siehe `files_modified`)
3. Initial-Inhalt pro Datei recherchebasiert füllen:
   - DPF-Zertifizierungs-Status auf https://www.dataprivacyframework.gov nachschauen
   - Vendor-Sub-Processor-Liste verlinken
   - Bezug zu konkreten Datentypen aus `PROCESSING-RECORDS.md` herstellen
4. Status aller 17 initial: ☐ Draft
5. `README.md` mit Übersichts-Tabelle (alle 17 + Status-Spalten)

## Verification

```bash
ls docs/legal/tia/ | wc -l   # erwartet: 19 (17 + template + README)
grep -l "Status: ☐ Draft" docs/legal/tia/*.md | wc -l  # erwartet: 17 (alle initial Draft)
```

`GDPR-AUDIT.md` Status-Tracking ORG-03 wird auf ✅ "Drafts erstellt" gesetzt; finaler ✅-Stempel erst nach Anwalts-Review.

## Out of scope

- Anwalts-Review (extern; separater Tracking-Schritt)
- DPF-Re-Zertifizierungs-Monitoring (sollte als jährlicher Cron-Task / Review gepflegt werden — nicht hier)
